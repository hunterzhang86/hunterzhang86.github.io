因为如果冲突比较大，事务需要回滚，应用层也会不断重试，这样反倒降低的性能

7. 常见问题

7.1. 如何优化分页查询？
大数据量的时候限制用户只能一页一页的往后翻

7.2. 可靠性实现？
redolog
binlog

7.3. 事务的可重复读是如何实现的？
给每行增加两个隐藏字段来实现MVVC(数据多版本并发控制)，两个列都用来存储事务的版本号
每开启一个新事务，事务的版本号就会递增
如果数据的trx_id落在低水位，可见
如果数据的trx_id落在高水位，不可见
如果数据在黄色区间，活跃事务ID数组

trx_id在数组中，说明未提交，不可见
trx_id不在数组中，说明已提交，可见

7.4. 读提交和可重复读实现区别？
可重复读在事务开始的时候创建一致性视图
读提交在语句执行的时候创建一致性试图

7.5. MySQL如何保证数据不丢失？
两阶段提交，先prepare，后commit
prepare其实做的一件事情就是刷盘
如果存在完整的prepare
如果binlog完整，则提交
可以保证数据和binlog的一致性
如果不完整，则回退
如果commit已经刷盘，则直接提交

7.6. MyISAM和InnoDB的区别？
MyISAM不支持事务
MyISAM不支持行锁
MyISAM支持FULLTEXT全文索引

7.7. count() count(1) count(col)的区别？
count() = count(1), 统计行不会忽略NULL
count(col), 统计行会忽略col=NULL的行

7.8. 可重复读隔离级别下，MVVC在数据的CRUD层面分别是如何实现的？
7.8.1. SELECT
读取创建版本小于或等于当前事务版本号，并且删除版本为空或大于当前事务版本号的记录。这样可以保证在读取之前记录是存在的
7.8.2. INSERT
将当前事务的版本号保存至行的创建版本号
7.8.3. UPDATE
新插入一行，并以当前事务的版本号作为新行的创建版本号，同时将原记录行的删除版本号设置为当前事务版本号
7.8.4. DELETE
将当前事务的版本号保存至行的删除版本号

8. 数据库锁

8.1. 全局锁

8.2. 表级锁

8.3. 行锁
8.3.1. S锁

8.3.2. X锁

都是针对索引加锁，如果索引失效，会升级为表锁
如果索引没命中，会退化成间隙锁

8.4. 间隙锁

间隙锁之间不会冲突，所以容易造成死锁
8.4.3. 原则
向左右查找第一个不满足条件的索引
当使用唯一索引的时候，如果等值刚好命中，不用向左右遍历找到不满足条件的索引
8.4.4. 作用
解决幻读的问题

8.5. 意向锁
8.5.1. 作用
解决上表锁的时候性能不够的问题，没有意向锁可能会一个一个判断行锁
8.5.2. 原则
只与表锁可能有互斥，不会与行锁进行比较

10. 同步机制

备库执行change master命令
备库执行start slave命令，会启动两个线程io_thread和sql_thread，其中io_thread负责与主库建立连接
主库将备库要读取的binlog发给备库，备库保存为relay log
备库读取relay log并执行
