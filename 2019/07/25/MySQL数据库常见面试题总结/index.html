MySQL 数据库常见面试题总结

By NoOne

Published 2019-07-25

MySQL是现阶段最流行的关系型数据库，也是后台开发面试过程中必问的知识点，本文主要对面试中经常出现的MySQL事务、锁和索引进行了总结，基本可以覆盖到面试过程中和MySQL相关的90%以上的知识点。

乐观锁和悲观锁

  • 乐观锁

    • 出发点

      • 认为数据一般情况下不会造成冲突

    • 优点

      • 并发好效率高

    • 缺点

      • ABA问题
        • 解决方法
          • 版本号
          • 时间戳

    • 使用场景

      • 读多写少，数据冲突不大的情况，考虑用户积分加减的场景
      • 因为如果冲突比较大，事务需要回滚，应用层也会不断重试，这样反倒降低的性能

    • SQL

      • update table set name = 'Aron', version = version + 1 where id = #{id} and version = #{version};

    • 使用方式

      • 先不上锁，但如果SQL执行失败应用层需要自己处理，比如重试

  • 悲观锁

    • 出发点

      • 认为数据比较大可能会发生冲突

    • 使用场景

      • 数据冲突比较大的情况

    • 使用方式

      • 每次在拿数据的时候都会上锁，这样别人想拿这个数据就会block直到它拿到锁

    • SQL

      • select ... id=1 for update

常见问题

  • 如何优化分页查询？

    • 大数据量的时候限制用户只能一页一页的往后翻

  • 可靠性实现？

    • redolog
    • binlog

  • 事务的可重复读是如何实现的？

    • 给每行增加两个隐藏字段来实现MVVC(数据多版本并发控制)，两个列都用来存储事务的版本号

    • 每开启一个新事务，事务的版本号就会递增

    • 如果数据的trx_id落在低水位，可见

    • 如果数据的trx_id落在高水位，不可见

    • 如果数据在黄色区间，活跃事务ID数组

    • trx_id在数组中，说明未提交，不可见

    • trx_id不在数组中，说明已提交，可见

  • 读提交和可重复读实现区别？

    • 可重复读在事务开始的时候创建一致性视图
    • 读提交在语句执行的时候创建一致性试图

  • MySQL如何保证数据不丢失？

    • 两阶段提交，先prepare，后commit
    • prepare其实做的一件事情就是刷盘
    • 如果存在完整的prepare
    • 如果binlog完整，则提交
    • 可以保证数据和binlog的一致性
    • 如果不完整，则回退
    • 如果commit已经刷盘，则直接提交

  • MyISAM和InnoDB的区别？

    • MyISAM不支持事务
    • MyISAM不支持行锁
    • MyISAM支持FULLTEXT全文索引

  • count(*) count(1) count(col)的区别？

    • count(*) = count(1), 统计行不会忽略NULL
    • count(col), 统计行会忽略col=NULL的行

  • 可重复读隔离级别下，MVVC在数据的CRUD层面分别是如何实现的？

    • SELECT
      • 读取创建版本小于或等于当前事务版本号，并且删除版本为空或大于当前事务版本号的记录。这样可以保证在读取之前记录是存在的
    • INSERT
      • 将当前事务的版本号保存至行的创建版本号
    • UPDATE
      • 新插入一行，并以当前事务的版本号作为新行的创建版本号，同时将原记录行的删除版本号设置为当前事务版本号
    • DELETE
      • 将当前事务的版本号保存至行的删除版本号

数据库锁

  • 全局锁

  • 表级锁

  • 行锁

  • S锁

  • X锁

    都是针对索引加锁，如果索引失效，会升级为表锁
    如果索引没命中，会退化成间隙锁

  • 间隙锁

    • select * from emp where empid > 100 for update;
    • 间隙锁之间不会冲突，所以容易造成死锁
    • 原则
      • 向左右查找第一个不满足条件的索引
      • 当使用唯一索引的时候，如果等值刚好命中，不用向左右遍历找到不满足条件的索引
    • 作用
      • 解决幻读的问题

  • 意向锁

    • 作用
      • 解决上表锁的时候性能不够的问题，没有意向锁可能会一个一个判断行锁
    • 原则
      • 只与表锁可能有互斥，不会与行锁进行比较

分布式事务

  • TCC

    • 柔性事务
    • 最终一致性
    • 作用在服务层

  • TXC

    • 分布式应用里面，多条数据库记录被同时修改带来的一致性问题
    • fascar

  • XA

    • 作用在资源层，依赖于数据库对XA协议的支持

同步机制

  • 备库执行 change master 命令
  • 备库执行 start slave 命令，会启动两个线程 io_thread 和 sql_thread，其中 io_thread 负责与主-库建立连接
  • 主库将备库要读取的 binlog 发给备库，备库保存为 relay log
  • 备库读取 relay log 并执行
