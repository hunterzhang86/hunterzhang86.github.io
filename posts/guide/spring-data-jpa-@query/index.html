Spring Data JPA @Query

 2023.3.4  2023.4.13  手册  4007  8 分钟  

1。概述

Spring Data 提供了许多方法来定义我们可以执行的查询。其中之一是 @Query 注释。

在本教程中，我们将演示如何在 Spring Data JPA 中使用 @Query 注释来执行 JPQL 和原生 SQL 查询。

我们还将展示如何在 @Query 注释不够时构建动态查询。

2。选择查询

为了为 Spring Data 存储库方法定义要执行的 SQL，我们可以 使用 @Query 注释对方法进行注释 --- 它的 value 属性包含要执行的 JPQL 或 SQL。

@Query 注释优先于命名查询，命名查询使用 @NamedQuery 注释或在 orm.xml 文件中定义。

将查询定义放在存储库中的方法上方而不是在我们的域模型中作为命名查询是一种很好的方法。存储库负责持久性，因此它是存储这些定义的更好地方。

2.1。 JPQL

默认情况下，查询定义使用 JPQL。

让我们看一个简单的存储库方法，它从数据库中返回活动的 User 实体：

1
2

@Query("SELECT u FROM User u WHERE u.status = 1")
集合<用户> findAllActiveUsers();

2.2。本国的

我们也可以使用原生 SQL 来定义我们的查询。我们所要做的就是将 nativeQuery 属性的值设置为 true 并在注解的 value 属性中定义原生 SQL 查询：

1
2
3
4

@询问（
  value = "SELECT * FROM USERS u WHERE u.status = 1",
  本机查询 = 真）
集合<用户> findAllActiveUsersNative();

3。在查询中定义顺序

我们可以将 Sort 类型的附加参数传递给具有 @Query 注释的 Spring Data 方法声明。它将被翻译成 ORDER BY 子句，然后传递给数据库。

3.1。 JPA 提供和派生方法的排序

对于我们开箱即用的方法，例如 findAll(Sort) 或通过解析方法签名生成的方法，我们只能使用对象属性来定义我们的排序：

1

userRepository.findAll(Sort.by(Sort.Direction.ASC, "name"));

现在假设我们要按名称属性的长度排序：

1

userRepository.findAll(Sort.by("LENGTH(name)"));

当我们执行上面的代码时，我们会收到一个异常：

1

org.springframework.data.mapping.PropertyReferenceException：找不到类型用户的属性长度（名称）！

3.2。 JPQL

当我们使用 JPQL 进行查询定义时，Spring Data 可以毫无问题地处理排序 --- 我们所要做的就是添加一个 Sort 类型的方法参数：

1
2

@Query(value = "从用户 u 中选择 u")
List<User> findAllUsers(排序排序);

我们可以调用这个方法并传递一个 Sort 参数，它将按照 User 对象的 name 属性对结果进行排序：

1

userRepository.findAllUsers(Sort.by("name"));

因为我们使用了 @Query 注释，所以我们可以使用相同的方法来获取 Users 按姓名长度排序的列表：

1

userRepository.findAllUsers(JpaSort.unsafe("LENGTH(name)"));

我们使用 JpaSort.unsafe() 创建 Sort 对象实例至关重要。

当我们使用：

1

Sort.by("LENGTH(姓名)");

那么我们将收到与上面看到的 findAll() 方法完全相同的异常。

当 Spring Data 发现使用 @Query 注释的方法存在不安全的 Sort 顺序时，它只是将排序子句附加到查询中 --- 它跳过检查要排序的属性是否属于域模型。

3.3。本国的

当 @Query 注释使用原生 SQL 时，无法定义 Sort。

如果我们这样做，我们将收到一个异常：

1

org.springframework.data.jpa.repository.query.InvalidJpaQueryMethodException：不能使用具有动态排序和/或分页的本机查询

正如异常所说，本机查询不支持排序。错误消息提示我们分页也会导致异常。

但是，有一种解决方法可以启用分页，我们将在下一节中介绍它。

4。分页

分页允许我们在 Page 中仅返回整个结果的子集。例如，在浏览网页上的多页数据时，这很有用。

分页的另一个优点是从服务器发送到客户端的数据量最小化。通过发送较小的数据，我们通常可以看到性能的提高。

4.1。 JPQL

在 JPQL 查询定义中使用分页很简单：

1
2

@Query(value = "SELECT u FROM User u ORDER BY id")
Page<User> findAllUsersWithPagination(Pageable pageable);

我们可以传递一个 PageRequest 参数来获取一页数据。

本机查询也支持分页，但需要一些额外的工作。

4.2。本国的

我们可以通过声明一个附加属性 countQuery 来为原生查询启用分页。

这定义了要执行的 SQL 来计算整个结果中的行数：

1
2
3
4
5

@询问（
  value = "SELECT * FROM Users ORDER BY id",
  countQuery = "SELECT count(*) FROM Users",
  本机查询 = 真）
Page<User> findAllUsersWithPagination(Pageable pageable);

4.3。 2.0.4 之前的 Spring Data JPA 版本

上述原生查询解决方案适用于 Spring Data JPA 2.0.4 及更高版本。

在该版本之前，当我们尝试执行这样的查询时，我们将收到与上一节中关于排序的相同的异常。

我们可以通过在查询中添加一个额外的分页参数来克服这个问题：

1
2
3
4
5

@询问（
  value = "SELECT * FROM Users ORDER BY id \n-- #pageable\n",
  countQuery = "SELECT count(*) FROM Users",
  本机查询 = 真）
Page<User> findAllUsersWithPagination(Pageable pageable);

在上面的例子中，我们添加

1

\n-- #pageable\n

作为分页参数的占位符。这告诉 Spring Data JPA 如何解析查询并注入可分页参数。此解决方案适用于 H2 数据库。

我们已经介绍了如何通过 JPQL 和原生 SQL 创建简单的选择查询。接下来，我们将展示如何定义附加参数。

5。索引查询参数

我们可以通过两种可能的方式将方法参数传递给我们的查询：索引参数和命名参数。

在本节中，我们将介绍索引参数。

5.1。 JPQL

对于 JPQL 中的索引参数，Spring Data 将按照它们在方法声明中出现的顺序将方法参数传递给查询：

1
2
3
4
5

@Query("SELECT u FROM User u WHERE u.status = ?1")
用户 findUserByStatus(Integer status);

@Query("SELECT u FROM User u WHERE u.status = ?1 and u.name = ?2")
用户 findUserByStatusAndName(Integer status, String name);

对于上述查询，status 方法参数将分配给索引为 1, 的查询参数，name 方法参数将分配给索引为 2 的查询参数。

5.2。本国的

本机查询的索引参数的工作方式与 JPQL 完全相同：

1
2
3
4

@询问（
  value = "SELECT * FROM Users u WHERE u.status = ?1",
  本机查询 = 真）
用户 findUserByStatusNative(Integer status);

在下一节中，我们将展示一种不同的方法：通过名称传递参数。

6。命名参数

我们还可以**使用命名参数将方法参数传递给查询。**我们使用存储库方法声明中的 @Param 注释来定义这些。

使用 @Param 注释的每个参数必须具有与相应 JPQL 或 SQL 查询参数名称匹配的值字符串。带有命名参数的查询更易于阅读，并且在需要重构查询的情况下更不容易出错。

6.1。 JPQL

如上所述，我们在方法声明中使用 @Param 注释将 JPQL 中由 name 定义的参数与方法声明中的参数匹配：

1
2
3
4

@Query("SELECT u FROM User u WHERE u.status = :status and u.name = :name")
用户 findUserByStatusAndNameNamedParams(
  @Param("status") 整数状态，
  @Param("name") 字符串名称);

请注意，在上面的示例中，我们将 SQL 查询和方法参数定义为具有相同的名称，但只要值字符串相同就不需要：

1
2
3

@Query("SELECT u FROM User u WHERE u.status = :status and u.name = :name")
用户 findUserByUserStatusAndUserName(@Param("status") 整数用户状态，
  @Param("name") 字符串用户名);

6.2。本国的

对于本机查询定义，与 JPQL 相比，我们通过名称将参数传递给查询的方式没有区别 --- 我们使用 @Param 注释：

1
2
3
4

@Query(value = "SELECT * FROM Users u WHERE u.status = :status and u.name = :name",
  本机查询 = 真）
用户 findUserByStatusAndNameNamedParamsNative(
  @Param("status") 整数状态，@Param("name") 字符串名称);

7。集合参数

让我们考虑 JPQL 或 SQL 查询的 where 子句包含 IN（或 NOT IN）关键字的情况：

1

SELECT u FROM User u WHERE u.name IN :names

在这种情况下，我们可以定义一个以Collection为参数的查询方法：

1
2

@Query(value = "SELECT u FROM User u WHERE u.name IN :names")
List<User> findUserByNameList(@Param("names") Collection<String> names);

由于参数是 Collection ，它可以与 List、HashSet 等一起使用。

接下来，我们将展示如何使用 @Modifying 注释修改数据。

8。使用 @Modifying 更新查询

我们可以通过在存储库方法中添加@Modifying 注释** 来**使用@Query 注释来修改数据库的状态。

8.1。 JPQL

与 select 查询相比，修改数据的存储库方法有两个不同之处 --- 它具有 @Modifying 注释，当然，JPQL 查询使用 update 而不是 select：

1
2
3
4

@修改
@Query("更新用户 u 设置 u.status = :status where u.name = :name")
int updateUserSetStatusForName(@Param("status") 整数状态，
  @Param("name") 字符串名称);

返回值定义了执行查询更新了多少行。索引参数和命名参数都可以在更新查询中使用。

8.2。本国的

我们也可以使用本机查询来修改数据库的状态。我们只需要添加 @Modifying 注释：

1
2
3
4

@修改
@Query(value = "更新用户设置 u.status = ? where u.name = ?",
  本机查询 = 真）
int updateUserSetStatusForNameNative（整数状态，字符串名称）；

8.3。插入

要执行插入操作，我们必须同时应用 @Modifying 并使用本机查询，因为 INSERT 不是 JPA 接口的一部分：

1
2
3
4
5
6
7

@修改
@询问（
  价值 =
    “插入用户（姓名，年龄，电子邮件，状态）值（：姓名，：年龄，：电子邮件，：状态）”，
  本机查询 = 真）
void insertUser(@Param("name") 字符串名称，@Param("age") 整数年龄，
  @Param("status") 整数状态，@Param("email") 字符串电子邮件）；

9。动态查询

通常，我们会遇到基于条件或数据集构建 SQL 语句的需求，这些条件或数据集的值仅在运行时才知道。在这些情况下，我们不能只使用静态查询。

9.1。动态查询示例

例如，假设我们需要从运行时定义的集合中选择所有电子邮件为 LIKE 的用户 --- email1 , email2 , ..., emailn：

1
2
3
4

SELECT u FROM User u WHERE u.email LIKE '%email1%'
    或 u.email LIKE '%email2%'
    ...
    或 u.email LIKE '%emailn%'

由于集合是动态构造的，我们无法在编译时知道要添加多少 LIKE 子句。

在这种情况下，我们不能只使用 @Query 注释，因为我们不能提供静态 SQL 语句。

相反，通过实现自定义复合存储库，我们可以扩展基础 JpaRepository 功能并提供我们自己的逻辑来构建动态查询。让我们来看看如何做到这一点。

9.2。自定义存储库和 JPA 标准 API

对我们来说幸运的是，Spring 提供了一种通过使用自定义片段接口来扩展基础存储库的方法。 然后我们可以将它们链接在一起以创建一个 复合存储库。

我们将从创建自定义片段接口开始：

1

    List<User> findUserByEmails(Set<String> emails);

此外，我们需要确保在类名中包含 Impl 后缀。 Spring 会将 UserRepositoryCustom 实现搜索为 UserRepositoryCustomImpl。由于片段本身不是存储库，因此 Spring 依靠这种机制来查找片段实现。

9.3。扩展现有存储库

请注意，第 2 节到第 7 节的所有查询方法都在 UserRepository 中。

所以，现在我们将通过扩展 UserRepository 中的新接口来集成我们的片段：

1

3

公共接口 UserRepository 扩展 JpaRepository<User, Integer>, UserRepositoryCustom {
    // 从第 2 节到第 7 节查询方法
}

9.4。使用存储库

最后，我们可以调用我们的动态查询方法：

1
2

4

Set<String> emails = new HashSet<>();
// 用任意数量的项目填充集合

我们已经成功创建了一个复合存储库并调用了我们的自定义方法。
