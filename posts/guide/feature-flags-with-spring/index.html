带有 Spring 的功能标志

 2023.2.21  2023.4.2  手册  3930  8 分钟  

1。概述

在本文中，我们将简要定义特性标志，并提出一种自以为是且实用的方法来在 Spring Boot 应用程序中实现它们。然后，我们将利用不同的 Spring Boot 特性深入研究更复杂的迭代。

我们将讨论可能需要功能标记的各种场景并讨论可能的解决方案。我们将使用比特币矿工示例应用程序来完成此操作。

2。功能标志

功能标志 - 有时称为功能切换 - 是一种允许我们启用或禁用应用程序的特定功能而无需修改代码或理想情况下重新部署我们的应用程序的机制。

根据给定功能标志所需的动态，我们可能需要全局配置它们，每个应用程序实例，或更细化——可能是每个用户或请求。

与软件工程中的许多情况一样，重要的是尝试使用最直接的方法来解决手头的问题，而不会增加不必要的复杂性。

功能标志是一种强大的工具，如果使用得当，可以为我们的系统带来可靠性和稳定性。但是，当它们被滥用或维护不足时，它们很快就会成为复杂性和令人头疼的根源。

有很多情况下特性标志可以派上用场：

基于主干的开发和重要功能

在基于主干的开发中，特别是当我们想要保持频繁集成时，我们可能会发现自己还没有准备好发布某个功能。功能标志可以派上用场，使我们能够继续发布，而无需在完成之前使我们的更改可用。

特定环境配置

我们可能会发现自己需要某些功能来为 E2E 测试环境重置我们的数据库。

或者，我们可能需要为非生产环境使用不同于生产环境的安全配置。

因此，我们可以利用功能标志在正确的环境中切换正确的设置。

A/B 测试

为同一个问题发布多个解决方案并衡量影响是一项引人注目的技术，我们可以使用特性标志来实现。

金丝雀发布

在部署新功能时，我们可能会决定逐步进行，从一小群用户开始，并在验证其行为的正确性时扩大其采用。功能标志使我们能够实现这一点。

在以下部分中，我们将尝试提供一种实用的方法来解决上述情况。

让我们分解不同的策略来进行特征标记，从最简单的场景开始，然后进入更精细和更复杂的设置。

3。应用程序级功能标志

如果我们需要处理前两个用例中的任何一个，应用程序级功能标志是一种让事情正常工作的简单方法。

一个简单的功能标志通常会涉及一个属性和一些基于该属性值的配置。

3.1。使用 Spring 配置文件的功能标志

在 Spring 中，我们可以利用配置文件。 方便地，配置文件使我们能够有选择地配置某些 bean。通过围绕它们的一些构造，我们可以快速为应用程序级功能标志创建一个简单而优雅的解决方案。

假设我们正在构建一个比特币挖矿系统。我们的软件已经投入生产，我们的任务是创建一个实验性的、改进的挖掘算法。

在我们的 JavaConfig 中，我们可以分析我们的组件：

公共类 ProfiledMiningConfig {

}

然后，使用之前的配置，我们只需要包含我们的配置文件即可选择加入我们的新功能。 有 [配置我们应用程序的方式](https://docs.spring.io/ spring-boot/docs/current/reference/html/boot-features-external-config.html）和[特别是启用配置文件]（https://docs.spring.io/spring-boot/docs/current/参考/html/boot-features-profiles.html）。同样，有 测试实用程序 让我们的生活更轻松。

只要我们的系统足够简单，我们就可以创建一个基于环境的配置来确定应用哪些功能标志以及忽略哪些功能标志。

假设我们有一个基于卡片而不是表格的新 UI，以及之前的实验性矿工。

我们希望在我们的验收环境 (UAT) 中启用这两个功能。我们可以在我们的 application.yml 文件中创建以下配置文件组：

春天：
  简介：
    团体：
      uat：实验矿工，ui-cards

有了前面的属性，我们只需要在 UAT 环境中启用 UAT 配置文件即可获得所需的功能集。当然，我们也可以在我们的项目中添加一个 application-uat.yml 文件，以包含我们的环境设置的其他属性。

在我们的例子中，我们希望 uat 配置文件也包括 experimental-miner 和 ui-cards.

注意：如果我们使用 2.4.0 之前的 Spring Boot 版本，我们将使用 UAT 配置文件特定文档中的 spring.profiles.include 属性来配置其他配置文件。与 spring.profiles.active 相比， 前者使我们能够以附加的方式包含配置文件。

3.2。使用自定义属性的功能标志

配置文件是完成工作的绝佳而简单的方法。但是，我们可能需要配置文件用于其他目的。或者，我们可能想要构建一个更加结构化的功能标志基础设施。

对于这些场景，自定义属性可能是一个理想的选择。

让我们利用 @ConditionalOnProperty 和我们的命名空间 重写我们之前的示例：

@配置
公共类 CustomPropsMiningConfig {

    @豆
    @ConditionalOnProperty(
      name = "features.miner.experimental",
      matchIfMissing = true)
    公共比特币矿工 defaultMiner() {
        返回新的 DefaultBitcoinMiner();
    }

    @豆
    @ConditionalOnProperty(
      名称 = "features.miner.experimental")
    公共比特币矿工实验矿工（）{
        返回新的 ExperimentalBitcoinMiner();
    }
}

前面的示例构建在 Spring Boot 的条件配置之上，并配置一个或另一个组件，具体取决于属性设置为 true 还是 false（或完全省略）。

结果与 3.1 中的结果非常相似，但现在，我们有了命名空间。拥有我们的命名空间允许我们创建有意义的 YAML/properties 文件：

#[...] 一些 Spring 配置

特征：
  矿工：
    实验：真
  用户界面：
    卡片：真实
    
#[...] 其他功能标志

此外，这个新设置允许我们为我们的功能标志添加前缀——在我们的例子中，使用 features 前缀*.*

这似乎是一个小细节，但随着我们的应用程序的增长和复杂性的增加，这个简单的迭代将帮助我们控制我们的功能标志。

让我们谈谈这种方法的其他好处。

3.4。公开功能配置

我们的比特币挖矿系统进行了 UI 升级，但尚未完全准备好。出于这个原因，我们决定对其进行功能标记。我们可能有一个使用 React、Angular 或 Vue 的单页应用程序。

无论采用何种技术，我们需要知道启用了哪些功能，以便我们可以相应地呈现我们的页面。

3.5。保持营地清洁

尽管这听起来很明显，但一旦我们深思熟虑地实施了我们的功能标志，那么在不再需要它们时保持纪律地摆脱它们同样重要。

第一个用例的功能标志——基于主干的开发和重要的功能——通常是短暂的。这意味着我们需要确保我们的 ConfigProperties、 我们的 Java 配置和我们的 YAML 文件保持干净和最新。

4。更多细粒度特征标志

有时我们会发现自己处于更复杂的场景中。对于 A/B 测试或金丝雀版本，我们以前的方法根本不够。

为了获得更细粒度的功能标志，我们可能需要创建我们的解决方案。这可能涉及自定义我们的用户实体以包含特定于功能的信息，或者可能扩展我们的 Web 框架。

然而，用功能标志污染我们的用户可能对每个人来说都不是一个吸引人的想法，而且还有其他解决方案。

作为替代方案，我们可以利用一些内置工具 例如 Togglz。这个工具增加了一些复杂性，但提供了一个很好的开箱即用的解决方案，并且 提供与 Spring Boot 的一流集成。

Togglz 支持不同的【激活策略】（https://www.togglz.org/documentation/activation-strategies.html）：

  1. **用户名：**与特定用户关联的标志
  2. **逐步推出：**为一定比例的用户群启用标志。这对于 Canary 版本很有用，例如，当我们想要验证功能的行为时
  3. **发布日期：**我们可以安排在特定日期和时间启用标志。这对于产品发布、协调发布或优惠和折扣可能很有用
  4. 客户端 IP： 基于客户端 IP 标记的功能。在将特定配置应用于特定客户时，这些可能会派上用场，因为他们有静态 IP
  5. 服务器 IP： 在这种情况下，服务器的 IP 用于确定是否应启用某个功能。这对于金丝雀版本也可能有用，其方法与逐步推出略有不同——比如当我们想要评估实例中的性能影响时
  6. **ScriptEngine：**我们可以基于[任意脚本]启用功能标志（https://docs.oracle.com/en/java/javase/11/docs/api/java.scripting/javax/script/ ScriptEngine.html)。这可以说是最灵活的选择
  7. **系统属性：**我们可以设置某些系统属性来确定功能标志的状态。这与我们使用最直接的方法所取得的效果非常相似
