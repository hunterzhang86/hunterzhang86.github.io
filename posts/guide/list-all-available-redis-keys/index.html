  1. 概述 ----------- 集合是几乎所有现代应用程序中常见的基本构建块。所以，毫无疑问，Redis 提供了多种【流行的数据结构】(https://redis.io/topics/data-types)如列表、集合、散列、排序集合等供我们使用.在本教程中，我们将学习如何有效地读取与特定模式匹配的所有可用 Redis 键。 2. 探索集合 ---------------------- 假设我们的应用程序使用 Redis 来存储有关不同运动中使用的球的信息。我们应该能够从 Redis 集合中看到有关每个可用球的信息。为简单起见，我们将数据集限制为仅三个球： * 重量为 160 克的板球 * 重量为 450 克的足球 * 重量为 270 克的排球 像往常一样，让我们首先通过工作来弄清楚我们的基础知识关于探索 Redis 集合的幼稚方法。 3. Naive Approach Using redis-cli --------------------------------- 在我们开始写Java代码之前先探索一下集合，我们应该对如何使用 redis-cli 接口有一个公平的想法。假设我们的 Redis 实例在端口 6379 上的 127.0.0.1 可用，以便我们使用命令行界面探索每种集合类型。 ### 3.1。链表首先，让我们借助 依赖 在本节中，我们将在我们的实现中为 Redis **使用 *[Jedis](/jedis-java-redis-client-library)* 客户端库**：xml < groupId>redis.clients jedis 3.2.0 ### 4.2. Redis 客户端 Jedis 库带有与 Redis-CLI 名称类似的方法。但是，建议我们**创建一个包装 Redis 客户端，它将在内部调用 Jedis 函数调用**。每当我们使用 Jedis 库时，我们必须牢记**单个 Jedis 实例不是线程安全的**。因此，要在我们的应用程序中获取 Jedis 资源，我们可以**使用 [*JedisPool*](/jedis-java-redis-client-library#Connection)，这是一个线程安全的网络连接池**。而且，由于我们不希望在应用程序的生命周期中的任何给定时间出现多个 Redis 客户端实例，我们应该根据 [singleton 设计模式](/java-singleton) 的原则创建我们的 *RedisClient* 类-双重检查锁定）。首先，让我们为我们的客户端创建一个私有构造函数，它会在创建 *RedisClient* 类的实例时在内部初始化 *JedisPool*：java private static JedisPool jedisPool;私有 RedisClient(String ip, int port) { try { if (jedisPool == null) { jedisPool = new JedisPool(new URI("http://" + ip + ":" + port)); } } } 返回实例； } 最后，让我们看看如何在 Jedis 的 *lrange 方法* 之上创建一个包装器方法：java public List lrange(final String key, final long start, final long stop) { try (Jedis jedis = jedisPool.getResource()) { return jedis.lrange(key, start, stop); } catch (Exception ex) { log.error("在 lrange 中捕获异常", ex); } 返回新的 LinkedList(); } 当然，我们可以按照相同的策略来创建其余的包装方法，例如 *lpush* 、 *hmset* 、 *hgetall* 、 *sadd* 、 *smembers* 、 *keys* 、 *zadd* , 和 *zrange*。 ### 4.3。分析我们可以用来**一次探索集合的所有 Redis 命令在最好的情况下自然会具有 O(n) 时间复杂度**。我们可能有点自由，称这种方法为幼稚。在 Redis 的实际生产实例中，在单个集合中拥有数千或数百万个键是很常见的。此外，[Redis 的单线程特性](https://redis.io/topics/latency#single-threaded-nature-of-redis) 带来了更多痛苦，我们的方法可能会灾难性地阻止其他更高优先级的操作。因此，我们应该指出，我们将我们的幼稚方法限制为仅用于调试目的。 5. 迭代器基础 ------------------ 我们幼稚实现的主要缺陷是我们请求 Redis 为我们的单个 fetch-query 提供所有结果一口气。为了克服这个问题，我们可以将原始提取查询分解为多个顺序提取查询，这些查询对整个数据集的较小块进行操作。假设我们有一本 1,000 页的书，我们应该阅读。如果我们遵循我们幼稚的方法，我们将不得不一口气读完这本大书，没有任何休息。这对我们的幸福来说是致命的，因为它会耗尽我们的能量并阻止我们进行任何其他优先级更高的活动。当然，正确的方法是通过多次阅读来完成这本书。在每个会话中，**我们从上一个会话中断的地方继续**——我们可以**使用页面书签跟踪我们的进度**。虽然这两种情况下的总阅读时间价值相当，但第二种方法更好，因为它给了我们喘息的空间。让我们看看如何使用基于迭代器的方法来探索 Redis 集合。 6. Redis Scan ------------- Redis 提供了几种扫描策略来使用基于光标的方法从集合中读取键，这在原理上类似于页面书签。 ### 6.1。扫描策略我们可以使用 *[Scan](https://redis.io/commands/scan)* 命令扫描整个键值集合存储。但是，如果我们想通过集合类型来限制我们的数据集，那么我们可以使用其中一种变体： * *[Sscan](https://redis.io/commands/sscan)* 可用于遍历集合 * [ *Hscan*](https://redis.io/commands/hscan) 帮助我们遍历哈希中的字段值对 * [*Zscan*](https://redis.io/commands/zscan) 允许通过存储在排序集中的成员进行迭代我们必须注意，我们**并不真正需要专门为链表设计的服务器端扫描策略**。这是因为我们可以使用 *[lindex](https://redis.io/commands/lindex)* 或 [*lrange*](https://redis.io/commands/lrange) 通过索引访问链表的成员） 命令。另外，我们可以找出元素的数量，并在一个简单的循环中使用 *lrange* 以小块迭代整个列表。让我们使用 *SCAN* 命令扫描字符串类型的键。 **要开始扫描，我们需要使用光标值为“0”**，匹配模式字符串为“ball\*”：``shell 127.0.0.1:6379> mset balls:cricket 160 balls:football 450 球：排球 270 OK 127.0.0.1:6379> SCAN 0 MATCH ball* COUNT 1 1) "2" 2) 1) "balls:cricket" 127.0.0.1:6379> SCAN 2 MATCH ball* COUNT 1 1) "3 " 2) 1) "balls:volleyball" 127.0.0.1:6379> SCAN 3 MATCH ball* COUNT 1 1) "0" 2) 1) "balls:football" 每完成一次扫描，我们都会得到下一个值在后续迭代中使用的光标。最终，当下一个光标值为“0”时，我们知道我们已经扫描了整个集合。 7. 使用 Java 进行扫描 --------------------- 至此，我们对我们的方法有了足够的了解，可以开始在 Java 中实现它了。 ### 7.1。扫描策略 如果我们深入了解 Jedis 类提供的核心扫描功能，我们会发现扫描不同集合类型的策略： java public ScanResult<String> scan(final String cursor, final ScanParams params);
