Java 和 Spring 中的事务简介

 2022.7.9  2022.10.20  手册  6628  14 分钟
  1. 简介 --------------- 在本教程中，我们将了解 Java 中事务的含义。从而我们将了解如何执行资源本地事务和全局事务。这也将允许我们探索在 Java 和 Spring 中管理事务的不同方法。 2. 什么是交易？ ------------------------- Java中的事务，一般是指[必须全部成功完成的一系列动作](/transactions-intro ）。因此，如果一个或多个操作失败，所有其他操作必须退出，而应用程序的状态保持不变。这对于确保应用程序状态的完整性永远不会受到损害是必要的。此外，这些事务可能涉及一个或多个资源，如数据库、消息队列，从而产生在事务下执行操作的不同方式。其中包括使用单个资源执行资源本地事务。或者，多个资源可以参与全局事务。 3. 资源本地事务 ------------------ 我们将首先探讨如何在 Java 中使用事务，同时使用个人资源。在这里，我们可能有多个单独的操作，我们使用数据库等资源执行。但是，我们可能希望它们作为一个统一的整体发生，就像在一个不可分割的工作单元中一样。换句话说，我们希望这些操作在单个事务下发生。在 Java 中，我们有多种方法可以访问和操作数据库等资源。因此，我们处理交易的方式也不尽相同。在本节中，我们将了解如何在 Java 中经常使用的这些库中使用事务。 ### 3.1。 JDBC Java 数据库连接 (JDBC) 是 Java 中的 API，它定义了如何访问 Java 中的数据库。不同的数据库供应商提供 JDBC 驱动程序，用于以与供应商无关的方式连接到数据库。因此，我们从驱动程序中检索 Connection 以对数据库执行不同的操作： JDBC 为我们提供了在事务下执行语句的选项。 Connection 的默认行为是自动提交。澄清一下，这意味着每一条语句都被视为一个事务，并在执行后立即自动提交。但是，如果我们希望在单个事务中捆绑多个语句，这也可以实现：java Connection connection = DriverManager.getConnection(CONNECTION_URL, USER, PASSWORD);尝试 { connection.setAutoCommit(false); JTA 事务管理 ----------------- 现在是时候了解我们如何使用 JTA 管理分布式事务了。分布式事务不是微不足道的解决方案，因此也具有成本影响。此外，我们可以从多个选项中选择将 JTA 包含在我们的应用程序中。因此，我们的选择必须考虑到整体应用程序架构和愿望。 ### 5.1。应用服务器中的 JTA 正如我们之前所见，JTA 架构依赖应用服务器来促进许多与事务相关的操作。它依赖服务器提供的关键服务之一是通过 JNDI 提供的命名服务。这是 XA 资源（如数据源）绑定和检索的地方。除此之外，我们可以选择如何管理应用程序中的事务边界。这在 Java 应用程序服务器中产生了两种类型的事务： * 容器管理的事务 ：顾名思义，这里的事务边界由应用程序服务器设置。这简化了 Enterprise Java Beans (EJB) 的开发，因为它不包括与事务划分相关的语句，并且仅依赖于容器来执行此操作。但是，这并没有为应用程序提供足够的灵活性。 * Bean 管理的事务：与容器管理的事务相反，在 bean 管理的事务中 EJB 包含定义事务分界的显式语句。这为标记事务边界的应用程序提供了精确控制，尽管代价是更复杂。在应用程序服务器的上下文中执行事务的主要缺点之一是应用程序与服务器紧密耦合。这对应用程序的可测试性、可管理性和可移植性有影响。这在微服务架构中更为深刻，其重点更多地放在开发与服务器无关的应用程序上。 ### 5.2。 JTA Standalone 我们在上一节中讨论的问题为创建不依赖于应用程序服务器的分布式事务解决方案提供了巨大的动力。在这方面，我们有多种选择，例如使用 Spring 的事务支持或使用 Atomikos 之类的事务管理器。让我们看看我们如何使用像 Atomikos 这样的事务管理器 来促进具有数据库和消息队列的分布式事务。分布式事务的关键方面之一是使用事务监视器列出和删除参与资源。 Atomikos 为我们解决了这个问题。我们所要做的就是使用 Atomikos 提供的抽象：java AtomikosDataSourceBean atomikosDataSourceBean = new AtomikosDataSourceBean(); Spring 中的事务支持------------------ 我们已经看到，**处理事务是相当涉及的任务包括大量样板代码**和配置。此外，每种资源都有自己处理本地事务的方式。在 Java 中，JTA 将我们从这些变化中抽象出来，但进一步带来了特定于提供商的细节和应用程序服务器的复杂性。 Spring 平台**为我们提供了一种更简洁的方式来处理事务，包括 Java 中的资源本地事务和全局事务**。这与 Spring 的其他优点一起为使用 Spring 处理事务创造了一个令人信服的案例。此外，使用 Spring 配置和切换事务管理器非常容易，它可以是服务器提供的，也可以是独立的。 Spring 通过使用事务代码为方法创建代理**，为我们提供了这种**无缝抽象。代理在 *TransactionManager* 的帮助下代表代码管理事务状态：![](https://www.baeldung.com/wp-content/uploads/2020/08/Spring-Transactions.jpg)这里的中心接口是*PlatformTransactionManager*，它有许多不同的实现可用。它提供对 JDBC（数据源）、JMS、JPA、JTA 和许多其他资源的抽象。 ### 6.1。配置 让我们看看如何配置 **Spring 以使用 Atomikos 作为事务管理器并为 JPA 和 JMS 提供事务支持**。我们将从定义 JTA 类型的 *PlatformTransactionManager* 开始：java @Bean public PlatformTransactionManager platformTransactionManager() throws Throwable { return new JtaTransactionManager( userTransaction(), transactionManager()); @Autowired JmsTemplate jmsTemplate; @Transactional(propagation = Propagation.REQUIRED) public void process(ENTITY, MESSAGE) { entityManager.persist(ENTITY); jmsTemplate.convertAndSend(DESTINATION, MESSAGE); } 上面的简单代码足以在 JTA 事务中允许数据库中的保存操作和消息队列中的发布操作。 * **程序化支持**虽然声明式支持非常优雅和简单，但它并没有为我们提供**更精确地控制事务边界的好处**。因此，如果我们确实需要实现这一点，Spring 提供了划定事务边界的编程支持：java @Autowired private PlatformTransactionManager transactionManager;公共无效进程（实体，消息）{ TransactionTemplate transactionTemplate = new TransactionTemplate(transactionManager); transactionTemplate.executeWithoutResult(status -> { entityManager.persist(ENTITY); jmsTemplate.convertAndSend(DESTINATION, MESSAGE); }); } ``` 所以，正如我们所见，我们必须使用可用的 PlatformTransactionManager 创建一个 TransactionTemplate。然后我们可以使用 TransactionTemplete 来处理全局事务中的一堆语句。 7. 事后思考---------------- 正如我们所见，处理事务，尤其是那些跨越多个资源的事务是复杂的。此外，事务本质上是阻塞的，这对应用程序的延迟和吞吐量有害。此外，使用分布式事务测试和维护代码并不容易，尤其是在事务依赖于底层应用服务器的情况下。所以，总而言之，如果可以的话，最好完全避免交易！但这远非现实。简而言之，在现实世界的应用程序中，我们确实经常有合法的交易需求。尽管可以重新考虑没有事务的应用程序架构，但这可能并不总是可能的。因此，在使用 Java 中的事务处理时，我们必须采用某些最佳实践以使我们的应用程序更好： * 我们应该采用的基本转变之一是使用独立的事务管理器而不是应用程序服务器提供的那些。仅此一项就可以大大简化我们的应用程序。此外，它非常适合云原生微服务架构。 * 此外，像 Spring 这样的抽象层可以帮助我们控制提供者的直接影响，比如 JPA 或 JTA 提供者。因此，这可以让我们在提供者之间切换，而不会对我们的业务逻辑产生太大影响。此外，它还从我们手中夺走了管理事务状态的低级职责。 * 最后，我们应该小心选择代码中的事务边界。由于事务是阻塞的，因此最好尽可能限制事务边界。如有必要，我们应该更喜欢对事务进行程序化控制而不是声明式控制。 8. 结论------------ 总而言之，在本教程中，我们讨论了Java 上下文中的事务。我们在 Java 中支持不同资源的单个资源本地事务。我们还介绍了在 Java 中实现全局事务的方法。此外，我们通过不同的方式来管理 Java 中的全局事务。此外，我们了解 Spring 如何让我们更轻松地在 Java 中使用事务。最后，我们介绍了在 Java 中处理事务时的一些最佳实践。 Java 底层 ### 通过 Learn Spring 课程开始使用 Spring 5 和 Spring Boot 2： >> 查看课程** 通用页脚横幅！使用 SPRING 构建您的 API - 书籍封面 ！使用 SPRING 构建您的 API - 图标 学习使用 Spring 构建 API **？[下载电子书](/休息API弹簧指南）
