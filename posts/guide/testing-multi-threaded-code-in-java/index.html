在 Java 中测试多线程代码

 2022.7.8  2022.10.20  手册  6513  13 分钟
  1. 简介 --------------- 在本教程中，我们将介绍测试并发程序的一些基础知识。我们将主要关注基于线程的并发性及其在测试中出现的问题。我们还将了解如何解决其中的一些问题并在 Java 中有效地测试多线程代码。 2. 并发编程------------- 并发编程是指我们将一大块计算分解成更小、相对独立的编程计算**。本练习的目的是同时运行这些较小的计算，甚至可能并行运行。虽然有几种方法可以实现这一点，但目标始终是更快地运行程序。 ### 2.1。线程和并发编程随着处理器的内核数量比以往任何时候都多，并发编程处于有效利用它们的最前沿。然而，事实仍然是并发程序更难设计、编写、测试和维护。因此，如果我们毕竟能够为并发程序编写有效的自动化测试用例，我们就可以解决这些问题中的很大一部分。那么，是什么让为并发代码编写测试如此困难？要理解这一点，我们必须了解我们如何在程序中实现并发。最流行的并发编程技术之一涉及使用线程。现在，线程可以是原生的，在这种情况下，它们是由底层操作系统调度的。我们还可以使用由运行时直接调度的所谓的绿色线程。 ### 2.2。测试并发程序的困难不管我们使用什么类型的线程，使它们难以使用的是线程通信。如果我们确实设法编写了一个涉及线程但没有线程通信的程序，那就没有比这更好的了！更现实的是，线程通常必须进行通信。有两种方法可以实现这一点——共享内存和消息传递。 与并发编程相关的大部分问题源于使用具有共享内存的本机线程。出于同样的原因，测试这样的程序也很困难。访问共享内存的多个线程通常需要互斥。我们通常通过一些使用锁的保护机制来实现这一点。但这可能仍会导致许多问题，例如竞争条件、活锁、死锁和线程饥饿，仅举几例。此外，这些问题是间歇性的，因为在本地线程的情况下线程调度是完全不确定的。因此，为并发程序编写能够以确定性方式检测这些问题的有效测试确实是一个挑战！ ### 2.3。线程交错剖析 我们知道，本机线程可以被操作系统以不可预知的方式调度。如果这些线程访问和修改共享数据，就会产生有趣的线程交错。虽然这些交织中的一些可能是完全可以接受的，但其他可能会使最终数据处于不希望的状态。让我们举个例子。假设我们有一个由每个线程递增的全局计数器。在处理结束时，我们希望这个计数器的状态与已执行的线程数完全相同：java private int counter;公共无效增量（）{计数器++； } 现在，在 Java 中增加原始整数不是原子操作。它包括读取值，增加它，最后保存它。虽然多个线程正在执行相同的操作，但它可能会产生许多可能的交错：![](https://www.baeldung.com/wp-content/uploads/2020/03/Screenshot-2020-03-27 -at-06.53.27-1024x235.png) 虽然这种特殊的交错会产生完全可接受的结果，但如何关于这个：[](/ wp-content/uploads/2020/03/Screenshot-2020-03-27-at-06.54.15.png）这不是我们所期望的。现在，想象一下数百个线程运行的代码比这复杂得多。这将导致线程交错的难以想象的方式。有几种方法可以编写代码来避免这个问题，但这不是本教程的主题。使用锁的同步是常见的同步之一，但它存在与竞争条件相关的问题。 3. 测试多线程代码 ------------------ 现在我们了解了测试多线程代码的基本挑战，我们将看看如何克服它们。我们将构建一个简单的用例，并尝试尽可能多地模拟与并发相关的问题。让我们从定义一个简单的类开始，该类保留可能的任何计数：java public class MyCounter { private int count;公共无效增量（）{ int temp = count;计数 = 温度 + 1； } // getter for count } 这是一段看似无害的代码，但不难理解它不是线程安全的。如果我们碰巧用这个类写了一个并发程序，它肯定是有缺陷的。此处测试的目的是识别此类缺陷。 ### 3.1。测试非并发部分根据经验，始终建议通过将代码与任何并发行为隔离来测试代码。这是为了合理地确定代码中没有其他与并发无关的缺陷。让我们看看如何做到这一点：java @Test public void testCounter() { MyCounter counter = new MyCounter(); for (int i = 0; i < 500; i++) { counter.increment(); } assertEquals(500, counter.getCount()); } 虽然这里没什么大不了的，但这个测试让我们相信它至少在没有并发的情况下可以工作。 ### 3.2。第一次尝试并发测试 让我们继续测试相同的代码，这次是在并发设置中。我们将尝试使用多个线程访问此类的同一个实例，并查看它的行为：java @Test public void testCounterWithConcurrency() throws InterruptedException { int numberOfThreads = 10; ExecutorService 服务 = Executors.newFixedThreadPool(10); CountDownLatch 锁存器 = new CountDownLatch(numberOfThreads); MyCounter 计数器 = new MyCounter(); for (int i = 0; i < numberOfThreads; i++) { service.execute(() -> { counter.increment(); latch.countDown(); }); }latch.await(); assertEquals(numberOfThreads, counter.getCount()); } 这个测试是合理的，因为我们正在尝试使用多个线程对共享数据进行操作。由于我们将线程数保持在较低水平，例如 10，我们会注意到它几乎一直都在通过。有趣的是，如果我们开始增加线程数，比如增加到 100，我们将看到测试在大多数情况下开始失败。 ### 3.3。更好的并发测试尝试 虽然前面的测试确实表明我们的代码不是线程安全的，但这个测试存在问题。此测试不是确定性的，因为底层线程以非确定性方式交错。对于我们的程序，我们真的不能依赖这个测试。我们需要的是一种控制线程交错的方法，以便我们可以用更少的线程以确定的方式揭示并发问题。我们将从调整我们正在测试的代码开始：java public synchronized void increment() throws InterruptedException { int temp = count;等待（100）；计数 = 温度 + 1； } 在这里，我们使方法同步，并在方法中的两个步骤之间引入了等待。 synchronized 关键字确保一次只有一个线程可以修改 count 变量，并且等待会在每个线程执行之间引入延迟。请注意，我们不一定要修改我们打算测试的代码。但是，由于我们可以影响线程调度的方法并不多，所以我们求助于这个。在后面的部分中，我们将看到如何在不更改代码的情况下做到这一点。现在，让我们像之前一样测试这段代码：java @Test public void testSummationWithConcurrency() throws InterruptedException { int numberOfThreads = 2; ExecutorService 服务 = Executors.newFixedThreadPool(10); CountDownLatch 锁存器 = new CountDownLatch(numberOfThreads); MyCounter 计数器 = new MyCounter(); for (int i = 0; i < numberOfThreads; i++) { service.submit(() -> { try { counter.increment(); } catch (InterruptedException e) { // 处理异常 } latch.countDown(); } ); }latch.await(); assertEquals(numberOfThreads, counter.getCount()); } 在这里，我们只用两个线程运行它，我们很有可能会得到我们一直缺少的缺陷。我们在这里所做的是尝试实现特定的线程交错，我们知道这会影响我们。虽然对演示很有用，但我们可能会发现这对实际目的没有用。 4. 可用的测试工具 -------------------------- 随着线程数量的增长，它们可能交错的方式数量呈指数增长。 只是不可能找出所有此类交错并对其进行测试。我们必须依靠工具来为我们承担相同或相似的工作。幸运的是，有几个可以让我们的生活更轻松。我们可以使用两大类工具来测试并发代码。第一个使我们能够对具有许多线程的并发代码产生相当高的压力。压力增加了罕见交错的可能性，从而增加了我们发现缺陷的机会。第二个使我们能够模拟特定的线程交错，从而帮助我们更确定地发现缺陷。 ### 4.1。 tempus-fugit tempus-fugit Java 库帮助我们轻松编写和测试并发代码。我们在这里只关注这个库的测试部分。我们之前看到，对具有多个线程的代码产生压力会增加发现与并发相关的缺陷的机会。虽然我们可以编写实用程序来自己产生压力，但 tempus-fugit 提供了方便的方法来实现同样的效果。让我们重新审视我们之前试图产生压力的相同代码，并了解我们如何使用 tempus-fugit 实现相同的效果：java public class MyCounterTests { @Rule public ConcurrentRule concurrently = new ConcurrentRule(); @Rule public RepeatingRule rule = new RepeatingRule();私有静态 MyCounter 计数器 = new MyCounter(); @Test @Concurrent(count = 10) @Repeating(repetition = 10) public void runsMultipleTimes() { counter.increment(); } @AfterClass public static void annotatedTestRunsMultipleTimes() 抛出 InterruptedException { assertEquals(counter.getCount(), 100); } } 在这里，我们使用 tempus-fugit 提供的两个 Rules。这些规则拦截测试并帮助我们应用所需的行为，例如重复和并发。因此，有效地，我们从十个不同的线程中重复被测操作十次。随着我们增加重复性和并发性，我们发现与并发相关的缺陷的机会也会增加。 ### 4.2。 Thread Weaver Thread Weaver 本质上是一个用于测试多线程代码的 Java 框架。我们之前已经看到线程交错是非常不可预测的，因此，我们可能永远无法通过常规测试发现某些缺陷。我们实际上需要的是一种控制交错并测试所有可能的交错的方法。在我们之前的尝试中，这已被证明是一项相当复杂的任务。让我们看看 Thread Weaver 如何在这里为我们提供帮助。 Thread Weaver 允许我们以多种方式交错执行两个单独的线程，而不必担心如何。它还使我们有可能对我们希望线程如何交错进行细粒度控制。让我们看看如何改进我们之前的幼稚尝试：java public class MyCounterTests { private MyCounter counter; @ThreadedBefore public void before() { counter = new MyCounter(); } @ThreadedMain public void mainThread() { counter.increment(); } @ThreadedSecondary public void secondThread() { counter.increment(); } @ThreadedAfter public void after() { assertEquals(2, counter.getCount()); } @Test public void testCounter() { new AnnotatedTestRunner().runTests(this.getClass(), MyCounter.class); } } 在这里，我们定义了两个试图增加计数器的线程。 Thread Weaver 将尝试在所有可能的交错场景中使用这些线程运行此测试。可能在其中一个交错中，我们会遇到缺陷，这在我们的代码中非常明显。 ### 4.3。 MultithreadedTC MultithreadedTC 是又一个用于测试并发应用程序的框架。它具有一个节拍器，用于对多线程中的活动顺序进行精细控制。它支持执行特定线程交错的测试用例。因此，理想情况下，我们应该能够确定性地测试单独线程中的每个重要交错。现在，对这个功能丰富的库的完整介绍超出了本教程的范围。但是，我们当然可以看到如何快速设置测试，为我们提供执行线程之间可能的交错。让我们看看如何使用 MultithreadedTC 更确定地测试我们的代码：java public class MyTests extends MultithreadedTestCase { private MyCounter counter; @Override public void initialize() { counter = new MyCounter(); } public void thread1() 抛出 InterruptedException { counter.increment(); } public void thread2() 抛出 InterruptedException { counter.increment(); } @Override public void finish() { assertEquals(2, counter.getCount()); } @Test public void testCounter() throws Throwable { TestFramework.runManyTimes(new MyTests(), 1000); } } 在这里，我们设置了两个线程来操作共享计数器并增加它。我们已将 MultithreadedTC 配置为使用这些线程执行此测试，以进行多达一千种不同的交错，直到它检测到一个失败。 ### 4.4。 Java jcstress OpenJDK 维护代码工具项目，以提供开发人员用于处理 OpenJDK 项目的工具。这个项目下有几个有用的工具，包括【Java并发压力测试（jcstress）】（https://openjdk.java.net/projects/code-tools/jcstress/）。这是作为一种实验性工具和测试套件开发的，用于研究 Java 中并发支持的正确性。虽然这是一个实验性工具，但我们仍然可以利用它来分析并发代码并编写测试来资助与之相关的缺陷。让我们看看如何测试本教程中迄今为止使用的代码。从使用的角度来看，这个概念非常相似：java @JCStressTest @Outcome(id = "1", expect = ACCEPTABLE_INTERESTING, desc = "One update lost.") @Outcome(id = "2", expect = ACCEPTABLE, desc = "两个更新。") @State public class MyCounterTests { private MyCounter counter; @Actor public void actor1() { counter.increment(); } @Actor public void actor2() { counter.increment(); } @Arbiter public void arbiter(I_Result r) { r.r1 = counter.getCount(); } } 在这里，我们用注释 State 标记了该类，这表明它包含由多个线程变异的数据。此外，我们使用了一个注解 Actor，它标记了保存由不同线程完成的操作的方法。最后，我们有一个标有注释 Arbiter 的方法，它基本上只在所有 Actor 访问过它时才访问状态。我们还使用注释 Outcome 来定义我们的期望。总体而言，设置非常简单直观。我们可以使用框架提供的测试工具来运行它，该工具会找到所有使用 JCStressTest 注释的类，并在多次迭代中执行它们以获得所有可能的交错。 5. 其他检测并发问题的方法 ------------------------------------------为并发代码编写测试很困难，但可能。我们已经看到了挑战以及克服这些挑战的一些流行方法。但是，我们可能无法仅通过测试来识别所有可能的并发问题——尤其是当编写更多测试的增量成本开始超过其收益时。因此，结合合理数量的自动化测试，我们可以使用其他技术来识别并发问题。这将增加我们发现并发问题的机会，而无需深入了解自动化测试的复杂性。我们将在本节中介绍其中的一些内容。 ### 5.1。静态分析 静态分析是指在没有实际执行的情况下对程序进行分析。现在，这样的分析有什么好处呢？我们将谈到这一点，但让我们首先了解它与动态分析的对比。到目前为止，我们编写的单元测试需要在他们测试的程序的实际执行中运行。这就是它们是我们主要称为动态分析的一部分的原因。请注意，静态分析绝不能替代动态分析。但是，它提供了一个宝贵的工具，可以在我们执行代码之前很久就检查代码结构并识别可能的缺陷。 静态分析利用大量经验丰富的模板和理解。虽然很可能只查看代码并与我们策划的最佳实践和规则进行比较，但我们必须承认这对于大型程序来说是不合理的。但是，有几种工具可以为我们执行这种分析。它们相当成熟，对于大多数流行的编程语言都有大量的规则。一种流行的 Java 静态分析工具是 FindBugs。 FindBugs 查找“错误模式”的实例。错误模式是一种经常出错的代码习惯用法。这可能是由于多种原因造成的，例如难以理解的语言特征、被误解的方法和被误解的不变量。 FindBugs 检查 Java 字节码中是否出现错误模式 而不实际执行字节码。这使用起来非常方便，运行速度也很快。 FindBugs 报告属于许多类别的错误，例如条件、设计和重复代码。它还包括与并发相关的缺陷。但是，必须注意 FindBugs 可能会报告误报。这些在实践中较少，但必须与手动分析相关联。 ### 5.2。模型检查模型检查是一种检查系统的有限状态模型是否满足给定规范的方法。现在，这个定义可能听起来太学术了，但请忍受一段时间！我们通常可以将计算问题表示为有限状态机。尽管这本身就是一个广阔的领域，但它为我们提供了一个模型，其中包含一组有限的状态和它们之间的转换规则，并具有明确定义的开始和结束状态。现在，规范定义了模型应该如何表现才能被认为是正确的。本质上，该规范包含模型所代表的系统的所有要求。获取规范的方法之一是使用由 Amir Pnueli 开发的时序逻辑公式。虽然在逻辑上可以手动执行模型检查，但这是非常不切实际的。幸运的是，这里有很多工具可以帮助我们。 Java 可用的此类工具之一是 Java PathFinder (JPF)。 JPF 是根据 NASA 多年的经验和研究而开发的。具体来说，JPF 是 Java 字节码的模型检查器。它以所有可能的方式运行程序，从而检查所有可能的执行路径上的属性违规，如死锁和未处理的异常。因此，它可以被证明在发现任何程序中与并发相关的缺陷方面非常有用。 6. 事后思考 ---------------- 到目前为止，最好避免与多线程代码相关的复杂性可能的。开发设计更简单、更容易测试和维护的程序应该是我们的首要目标。我们必须同意，现代应用程序通常需要并发编程。但是，我们可以在开发并发程序时采用一些最佳实践和原则，这可以让我们的生活更轻松。在本节中，我们将介绍其中一些最佳实践，但我们应该记住，这个列表还远未完成！ ### 6.1。降低复杂性 复杂性是一个即使没有任何并发元素也会使测试程序变得困难的因素。面对并发，这只会更加复杂。不难理解为什么更简单和更小的程序更容易推理，因此更容易进行有效测试。这里有几种可以帮助我们的最佳模式，例如 SRP（单一职责模式）和 KISS（保持愚蠢简单），仅举几例。现在，虽然这些并没有直接解决为并发代码编写测试的问题，但它们使这项工作更容易尝试。 ### 6.2。考虑原子操作原子操作是完全独立运行的操作。因此，可以简单地避免预测和测试交织的困难。比较和交换就是这样一种广泛使用的原子指令。简而言之，它将内存位置的内容与给定值进行比较，并且只有当它们相同时，才会修改该内存位置的内容。大多数现代微处理器都提供该指令的一些变体。 Java 提供了一系列原子类，如 AtomicInteger 和 AtomicBoolean，提供了下面的比较和交换指令的好处。 ### 6.3。拥抱不变性 在多线程编程中，可以更改的共享数据总是为错误留下空间。不变性是指实例化后无法修改数据结构的情况。这是并发程序的天作之合。如果对象的状态在创建后不能改变，竞争线程不必对它们申请互斥。这极大地简化了并发程序的编写和测试。但是，请注意，我们可能并不总是有选择不变性的自由，但我们必须在可能的情况下选择它。 ### 6.4。避免共享内存 大多数与多线程编程相关的问题都可以归因于我们在竞争线程之间共享内存这一事实。如果我们能摆脱他们怎么办！好吧，我们仍然需要一些机制让线程进行通信。 并发应用程序的替代设计模式为我们提供了这种可能性。流行的一种是 Actor 模型，它将 Actor 规定为并发的基本单元。在这个模型中，参与者通过发送消息进行交互。 Akka 是一个用 Scala 编写的框架，它利用 Actor Model 来提供更好的并发原语。 7. 结论---------- 在本教程中，我们介绍了与并发编程相关的一些基础知识。我们特别详细地讨论了 Java 中的多线程并发。在测试此类代码时，我们经历了它给我们带来的挑战，尤其是在使用共享数据时。此外，我们还介绍了一些可用于测试并发代码的工具和技术。我们还讨论了避免并发问题的其他方法，包括自动化测试之外的工具和技术。最后，我们介绍了一些与并发编程相关的编程最佳实践。本文的源代码可以在 GitHub 中找到。通用底部 ### 通过 Learn Spring 课程开始使用 Spring 5 和 Spring
