Flyway 回调指南

 2023.2.21  2023.4.15  手册  2029  5 分钟  

1。介绍

Flyway 库允许我们通过跟踪存储为 SQL 源代码的更改来对数据库进行版本控制。 每组更改都称为迁移。

使用包括 migrate、clean、info、validate、 baseline 和 repair 在内的一组命令按顺序将单个迁移应用到数据库。它们根据目标数据库的当前版本以受控方式应用。

虽然迁移通常足以涵盖大多数用例，但有许多场景非常适合回调。

在本文中，我们将使用 Flyway 回调来挂钩它提供的各种命令的生命周期。

2。用例场景

我们可能有一个非常具体的要求，需要回调提供的那种灵活性。以下是一些可能的用例：

  • 重建物化视图 -- 当我们应用影响这些视图的基表的迁移时，我们可能希望重建物化视图。 SQL 回调非常适合执行这种类型的逻辑
  • 刷新缓存 -- 也许我们有一个迁移，它修改了恰好被缓存的数据。我们可以使用回调来刷新缓存，确保我们的应用程序从数据库中提取新数据
  • 调用外部系统 -- 使用回调，我们可以使用任意技术调用外部系统。例如，我们可能想要发布事件、发送电子邮件或触发服务器重启

3。支持的回调

每个可用的 Flyway 命令都有一个对应的 before 和 after 回调事件。有关这些命令的更多信息，请参阅 我们的主要 Flyway 文章，或 官方文档。

  • BEFORE_ 事件在操作执行之前被触发。
  • AFTER_ 事件在操作成功后触发。这些 after 事件还有一些更细化的事件：
    • ERROR 等价物在操作失败后触发。
    • OPERATION_FINISH 事件在操作完成后触发。
  • migrate 和 undo 也有 _EACH 事件，它会为每个单独的迁移触发。 migrate 和 undo 命令具有这些额外的回调，因为运行这些命令通常会导致执行许多迁移。

*完整的回调事件列表可以在Event类中找到。 *

例如， clean 命令的回调事件是 BEFORE_CLEAN 和 AFTER_CLEAN 。 Flyway 在 clean 命令执行前后立即触发它们。

回顾我们在介绍中讨论的内容，可用的命令有：migrate、clean、info、validate、 baseline 和 repair。

Flyway 的作者提供了这些额外的钩子，让我们可以在 Flyway 使用的最高粒度级别（即单独的迁移）上控制自定义回调逻辑。

5。回调

Flyway 使我们能够使用两种不同的方法（Java 或 SQL）创建回调。前者是最灵活的。它为我们提供了执行任意代码的自由。

后者让我们直接与数据库交互。

5.1。 Java 回调

Java API 契约在 Callback 接口中定义。

在最简单的情况下，要创建自定义回调，我们需要实现 Callback 接口，如 ExampleFlywayCallback:

公共类 ExampleFlywayCallback 实现回调 {

    私有最终日志 log = LogFactory.getLog(getClass());

    @覆盖
    公共布尔支持（事件事件，上下文上下文）{
        返回事件 == Event.AFTER_EACH_MIGRATE；
    }

    @覆盖
    公共布尔canHandleInTransaction（事件事件，上下文上下文）{
        返回真；
    }

    @覆盖
    公共无效句柄（事件事件，上下文上下文）{
        如果（事件 == Event.AFTER_EACH_MIGRATE）{
            log.info("> afterEachMigrate");
        }
    }

    @覆盖
    公共字符串 getCallbackName() {
        返回 ExampleFlywayCallback.class.getSimpleName();
    }
}

5.2。 SQL 回调

SQL 回调合约是通过使用包含在配置为 locations(s) 的目录中的特定名称的文件来定义的。 Flyway 将在其配置的 locations 中查找 SQL 回调文件并相应地执行它们.

例如，在 migrate 命令执行期间，配置为 location 的目录中名为 beforeEachMigrate.sql 的文件将在每个迁移脚本之前运行。
