Java 中的清洁编码

 2022.7.7  2022.10.7  手册  6521  14 分钟
  1. 概述----------- 在本教程中，我们将介绍干净的编码原则。我们还将了解为什么干净的代码很重要以及如何在 Java 中实现这一点。此外，我们将看看是否有任何工具可以帮助我们。 2. 什么是干净代码？ ---------------------- 所以，在我们深入了解干净代码的细节之前，让我们先了解一下干净代码是什么意思。老实说，这个问题不可能有一个好的答案。在编程中，一些关注点跨越并因此产生了一般原则。但是，每种编程语言和范式都呈现出它们自己的一组细微差别，这要求我们采用合适的实践。从广义上讲，干净的代码可以概括为任何开发人员都可以轻松阅读和更改的代码。虽然这听起来像是对概念的过度简化，但我们将在本教程后面看到它是如何构建的。在我们听到有关干净代码的任何地方，我们可能都会提到 Martin Fowler。以下是他在其中一个地方描述干净代码的方式： > 任何傻瓜都可以编写计算机可以理解的代码。优秀的程序员编写人类可以理解的代码。 3. 我们为什么要关心干净的代码？ --------------------------------------- 编写干净的代码同样是个人习惯问题因为这是技巧问题。作为开发人员，随着时间的推移，我们通过经验和知识而成长。但是，我们必须问，我们到底为什么要投资开发干净的代码？我们知道其他人可能会发现阅读我们的代码更容易，但这种激励是否足够？让我们来了解一下！干净的编码原则帮助我们实现了许多与我们打算生产的软件相关的理想目标。让我们通过它们来更好地理解它： * 可维护的代码库：我们开发的任何软件都有高效的生命周期，在此期间将需要更改和一般维护。干净的代码可以帮助开发易于随时间更改和维护的软件。 * 更轻松的故障排除：由于各种内部或外部因素，软件可能会出现意外行为。就修复和可用性而言，它可能通常需要快速周转。使用简洁编码原则开发的软件更容易解决问题。 * 更快的入职：软件在其生命周期内将看到许多开发人员创建、更新和维护它，开发人员会在不同的时间点加入。这需要更快的入职以保持高生产力，干净的代码有助于实现这一目标。 4. 干净代码的特点-------------------------------- 用干净的编码原则编写的代码库表现出一些设置它们的特性分开。让我们来看看其中的一些特征： * Focused ：应该编写一段代码来解决特定问题。它不应该做任何与解决给定问题完全无关的事情。这适用于代码库中的所有抽象级别，如方法、类、包或模块。 * 简单：这是迄今为止最重要且经常被忽略的干净代码的特征。软件设计和实现必须尽可能简单，这可以帮助我们达到预期的结果。代码库中不断增加的复杂性使它们容易出错并且难以阅读和维护。 * Testable ：干净的代码，虽然简单，但必须解决手头的问题。它必须直观且易于测试代码库，最好以自动化方式。这有助于建立代码库的基线行为，并使其更容易在不破坏任何内容的情况下进行更改。这些是帮助我们实现上一节中讨论的目标的原因。与后来的重构相比，考虑到这些特征开始开发是有益的。这会降低软件生命周期的总拥有成本。 5. Java 中的 Clean Coding ----------- 现在我们已经了解了足够的背景知识，让我们看看如何将 Clean Coding 原则融入 Java . Java 提供了许多可以帮助我们编写干净代码的最佳实践。我们将它们分类在不同的类别中，并了解如何使用代码示例编写干净的代码。 ### 5.1。项目结构 虽然 Java 不强制实施任何项目结构，遵循一致的模式来组织我们的源文件、测试、配置、数据和其他代码工件总是有用的。 Maven 是一种流行的 Java 构建工具，规定了特定的项目结构。虽然我们可能不使用 Maven，但坚持约定总是好的。让我们看看 Maven 建议我们创建的一些文件夹： * src/main/java：用于源文件 * src/main/resources：用于资源文件，如属性 * src/test/java：用于测试source files * src/test/resources：用于测试资源文件，比如 properties关于我们的需求和观众。 ### 5.2。命名约定 遵循命名约定可以大大提高我们的代码可读性和可维护性。 Spring 的创建者 Rod Johnson 在 Spring 中强调 命名约定的重要性：> "...如果你知道什么确实，你很有可能猜到它的 Spring 类或接口的名称......" Java [规定了一组规则](https://www.oracle.com/technetwork/java/codeconventions-135099 .html) 在 Java 中命名任何东西时要遵守。一个格式良好的名称不仅有助于阅读代码，而且还传达了很多关于代码意图的信息。让我们举一些例子： * 类：就面向对象的概念而言，类是通常代表现实世界对象的对象的蓝图。因此，使用名词来命名充分描述它们的类是有意义的： java public class Customer { } * Variables：Java 中的变量捕获从类创建的对象的状态。变量的名称应该清楚地描述变量的意图：java public class Customer { private String customerName; } * Methods ：Java 中的方法始终是类的一部分，因此通常表示对从类创建的对象的状态进行的操作。因此 使用动词命名方法很有用: java public class Customer { private String customerName; public String getCustomerName() { return this.customerName; } } ``` 虽然我们只讨论了如何在 Java 中命名标识符，但请注意，还有其他最佳实践，如驼峰式大小写，我们应该注意其可读性。还可以有更多与命名接口、枚举、常量相关的约定。 ### 5.3。源文件结构 源文件可以包含不同的元素。虽然 Java **编译器强制执行某些结构，但很大一部分是流动的**。但是遵守将元素放置在源文件中的特定顺序可以显着提高代码的可读性。有多种流行的风格指南可以从中获取灵感，例如 [Google](https://google.github.io/styleguide/javaguide.html) 和 [Spring](https://github.com/) spring-projects/spring-framework/wiki/Code-Style）。让我们看看源文件中元素的典型顺序应该是怎样的： * 包语句 * 导入语句 * 所有静态导入 * 所有非静态导入 * 正是一个顶级类 * 类变量 * 实例变量 * 构造函数 * 方法除了以上，**方法可以根据它们的功能或范围进行分组**。没有一个好的约定，想法应该是**一次决定，然后始终如一地遵循。**让我们看一个格式良好的源文件：```java # /src/main/java/com/baeldung/application /entity/Customer.java 包 com.baeldung.application.entity；导入 java.util.Date；公共类客户{私人字符串客户名称；私人日期加入日期；公共客户（字符串客户名称）{ this.customerName = 客户名称； this.joiningDate = new Date(); } public String getCustomerName() { return this.customerName; } 公共日期 getJoiningDate() { 返回 this.joiningDate; } } ``` ### 5.4.空白我们都知道，与大块文本相比，短段落更容易阅读和理解。在阅读代码方面也没有太大的不同。放置良好且一致的空格和空行可以增强代码的可读性。这里的想法是在代码中引入逻辑分组，这可以帮助组织思维过程，同时尝试通读它。这里没有一个单一的规则可以采用，而是一套通用的指导方针和保持可读性的内在意图： * 开始静态块、字段、构造函数和内部类之前的两个空行 * 方法之后的一个空行多行签名 * 一个空格分隔保留关键字，如 if、for、catch 和左括号 * 一个空格分隔保留关键字，如 else、catch 和右括号 这里的列表并不详尽，但应该让我们了解一下向。 ### 5.5。缩进虽然很微不足道，但几乎所有开发人员都会保证**缩进良好的代码更容易阅读和理解**。 Java 中的代码缩进没有单一的约定。这里的关键是要么采用一种流行的约定，要么定义一个私有的约定，然后在整个组织中始终如一地遵循它。让我们看看一些重要的缩进标准： * 一个典型的最佳实践是使用四个空格，一个缩进单位。请注意，一些指南建议使用制表符而不是空格。虽然这里没有绝对的最佳实践，但关键仍然是一致性！ * 通常，行长应该有一个上限，但由于开发人员现在使用更大的屏幕，这可以设置为高于传统的 80。 * 最后，由于许多表达式不能放在一行中，因此我们必须一致地对它们进行拆分： * 在逗号之后中断方法调用 * 在运算符之前中断表达式 * 缩进换行以提高可读性（我们在 Baeldung 更喜欢两个空格）让我们看一个例子：```java List<String> customerIds = customer.stream() .map(customer -> customer.getCustomerId()) .collect(Collectors.toCollection(ArrayList::new)); ``` ### 5.6.方法参数参数对于方法按照规范工作是必不可少的。但是，**一长串参数可能会让某人难以阅读和理解代码**。那么，我们应该在哪里画线呢？让我们了解可能对我们有帮助的最佳实践： * 尝试限制方法接受的参数数量，三个参数可能是一个不错的选择 * 如果方法需要超过推荐的参数，请考虑 [refactoring](/cs/refactoring) 该方法，通常一个长的参数列表也表明该方法可能正在做多种事情 *我们可以考虑将参数捆绑到自定义类型中，但必须注意不要将不相关的参数转储到单个类型中 *最后，虽然我们应该使用这个建议来判断代码的可读性，我们千万不要学究气让我们看一个例子：```java public boolean setCustomerAddress(String firstName, String lastName, String streetAddress, String city, String zipCode, String state, String country, String phoneNumber) { } // 这可以重构如下以增加可读性 public boolean setCustomerAddress(Address address) { } ``` ### 5.7.硬编码代码中的硬编码值通常会导致多种副作用。例如，**它可能导致重复，从而使更改更加困难**。如果值需要是动态的，它通常会导致不良行为。在大多数情况下，可以通过以下方式之一重构硬编码值： * 考虑用 Java 中定义的常量或枚举替换 * 否则，用在类级别或单独的类文件中定义的常量替换 * 如果可能，替换为可以从配置或环境中选择的值让我们看一个例子：```java private int storeClosureDay = 7; // 这可以重构为使用来自 Java private int storeClosureDay = DayOfWeek.SUNDAY.getValue() 的常量 ``` 同样，没有严格的指导方针来遵守。但我们必须认识到，有些人以后需要阅读和维护这段代码。我们应该选择一个适合我们的约定并保持一致。 ### 5.8。代码注释 [代码注释](/cs/clean-code-comments) 可以**有益于阅读代码以了解重要方面**。同时，应注意**不要在评论中包含明显的内容**。这可能会使评论变得臃肿，从而难以阅读相关部分。 Java 允许两种类型的注释：实现注释和文档注释。它们也有不同的用途和不同的格式。让我们更好地理解它们： * 文档/JavaDoc 注释 * 这里的受众是代码库的用户 * 这里的细节通常是免费的，更多地关注规范 * 通常独立于代码库有用 * 实现/块注释 * 这里的受众开发人员是否在代码库上工作 * 这里的细节是特定于实现的 * 通常与代码库一起使用 那么，我们应该如何优化使用它们，以便它们有用和上下文相关？ * 注释应该只是对代码的补充，如果没有注释我们无法理解代码，也许我们需要重构它 * 我们应该很少使用块注释，可能是为了描述重要的设计决策 * 我们应该在大多数情况下使用 JavaDoc 注释我们的类、接口、公共和受保护的方法 *所有注释都应该格式正确，并带有适当的缩进以提高可读性让我们看一个有意义的文档注释示例：java /** 此方法旨在添加一个新的客户的地址。 * 但是请注意，每个邮政编码只允许一个地址。因此，这将覆盖具有 * 相同邮政编码的任何先前地址。 * * @param address 要为现有客户添加的地址 / / * 此方法利用 equals * 方法的自定义实现，以避免重复具有相同邮政编码的地址。 / public addCustomerAddress(地址地址) { } ### 5.9.日志 任何曾经接触生产代码进行调试的人都渴望在某个时间点获得更多日志。 **日志的重要性在一般开发和维护中怎么强调都不过分**。 Java中有很多用于日志记录的库和框架，包括SLF4J、Logback。虽然他们使代码库中的日志记录变得非常简单，但必须注意记录最佳实践。否则完成的日志记录可以证明是维护的噩梦，而不是任何帮助。让我们来看看其中的一些最佳实践： * 避免过多的日志记录，考虑哪些信息可能有助于故障排除 * 明智地选择日志级别，我们可能希望在生产环境中选择性地启用日志级别 * 使用上下文数据非常清晰和具有描述性日志消息 * 使用外部工具来跟踪、聚合、过滤日志消息以加快分析速度让我们看一个具有正确级别的描述性日志记录示例：java logger.info(String.format("一个新客户已创建客户 ID: %s", id)); ``` 6. 这就是全部吗？ --------------------- 虽然上一节重点介绍了几种代码格式约定，但这些并不是我们应该知道和关心的唯一内容。一个可读和可维护的代码可以受益于随着时间的推移积累的大量额外的最佳实践。随着时间的推移，我们可能会遇到它们作为有趣的首字母缩略词。他们基本上将学习作为一个或一组原则来帮助我们编写更好的代码。但是，请注意，我们不应该仅仅因为它们存在就遵循它们。大多数时候，它们提供的好处与代码库的大小和复杂性成正比。在采用任何原则之前，我们必须访问我们的代码库。更重要的是，我们必须与他们保持一致。 ### 6.1。 SOLID SOLID 是一个助记词首字母缩写词，它借鉴了 它提出的五个原则 用于编写可理解和可维护的软件： * 单一责任原则：我们定义的每个 接口、类或方法都应该具有明确定义的目标。从本质上讲，理想情况下它应该做一件事并做好。这有效地导致更小的方法和类也是可测试的。 * 开闭原则：理想情况下，我们编写的代码应该是对扩展开放，对修改关闭。这实际上意味着一个类应该以不需要修改它的方式编写。但是，它应该允许通过继承或组合进行更改。 * Liskov 替换原则：该原则规定每个子类或派生类都应该可以替换它们的父类或基类。这有助于减少代码库中的耦合，从而提高跨平台的可重用性。 * 接口隔离原则：实现接口是为我们的类提供特定行为的一种方式。但是，一个类必须不需要实现它不需要的方法。这需要我们做的是定义更小、更集中的接口。 * 依赖倒置原则：根据这个原则，类应该只依赖于抽象而不是它们的具体实现。这实际上意味着一个类不应该负责为其依赖项创建实例。相反，应该将此类依赖项注入到类中。 ### 6.2。 DRY & KISS DRY 代表“Don's Repeat Yourself”。该原则规定一段代码不应在软件中重复。这一原则背后的基本原理是减少重复并提高可重用性。但是，请注意，我们应该谨慎地采用这种字面意思。一些重复实际上可以提高代码的可读性和可维护性。 KISS 代表“保持简单，愚蠢”。这个原则表明我们应该尽量保持代码尽可能简单。这使得随着时间的推移易于理解和维护。遵循前面提到的一些原则，如果我们保持我们的类和方法集中且小，这将导致更简单的代码。 ### 6.3。 TDD TDD 代表“测试驱动开发”。这是一种编程实践，要求我们仅在自动化测试失败时编写任何代码。因此，我们必须从自动化测试的设计开发开始。在 Java 中，有几个框架可以编写自动化单元测试，例如 JUnit 和 TestNG。这种做法的好处是巨大的。这导致软件始终按预期工作。由于我们总是从测试开始，我们以小块的方式逐步添加工作代码。此外，我们仅在新测试或任何旧测试失败时添加代码。这意味着它也导致了可重用性。 7. 帮助工具 ----------------- 编写干净的代码不仅仅是一个原则和实践的问题，而是一种个人习惯。随着我们的学习和适应，我们往往会成长为更好的开发人员。但是，为了在大型团队中保持一致性，我们还必须进行一些强制执行。代码审查一直是保持一致性并通过建设性反馈帮助开发人员成长的好工具。但是，我们不必在代码审查期间手动验证所有这些原则和最佳实践。 Freddy Guime [来自 Java OffHeap](https://www.javaoffheap.com/2019/10/episode-47-microsoft-flexing-its-java-muscle-javafx-is-alive-and-well-and-would- you-approve-my-low-quality-pr.html) 讨论了自动执行某些质量检查以始终以一定的代码质量阈值结束的价值。 Java 生态系统中有几个可用的工具，它们至少可以让代码审查员承担其中的一些责任。让我们看看其中一些工具是什么： * 代码格式化程序：大多数流行的 Java 代码编辑器，包括 Eclipse 和 IntelliJ，都允许自动代码格式化。我们可以使用默认的格式规则，自定义它们，或者用自定义格式规则替换它们。这需要处理许多结构代码约定。 * 静态分析工具：Java 有多种静态代码分析工具，包括SonarQube、Checkstyle、PMD 和[SpotBugs](https: //spotbugs.github.io/)。他们有一套丰富的规则，可以按原样使用或为特定项目定制。它们非常擅长检测许多 代码异味，例如违反命名约定和资源泄漏。 8. 结论 ------------- 在本教程中，我们已经了解了干净编码原则和干净代码所展示的特征的重要性。我们看到了如何在实践中采用其中的一些原则，这些原则是用 Java 开发的。我们还讨论了其他有助于保持代码可读性和可维护性的最佳实践。最后，我们讨论了一些可以帮助我们完成这项工作的工具。总而言之，重要的是要注意所有这些原则和实践都是为了使我们的代码更干净。这是一个更主观的术语，因此必须根据上下文进行评估。虽然有许多规则可供采用，但我们必须意识到我们的成熟度、文化和要求。我们可能必须定制或为此设计一套新的规则。但是，无论情况如何，重要的是要在整个组织中保持一致以获得收益。
