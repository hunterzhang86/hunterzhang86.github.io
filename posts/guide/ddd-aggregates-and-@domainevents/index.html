DDD 聚合和@DomainEvents

 2023.4.14  2023.4.24  手册  3562  8 分钟  

1。概述

在本教程中，我们将解释如何使用 @DomainEvents 注释和 AbstractAggregateRoot 类来方便地发布和处理聚合产生的领域事件——这是领域驱动设计中的关键战术设计模式之一。

聚合接受业务命令，这通常会导致产生与业务领域相关的事件——领域事件。

如果您想了解有关 DDD 和聚合的更多信息，最好从 Eric Evans 的 原著 开始。还有一个很棒的 [关于有效聚合设计的系列] (http://dddcommunity.org/wp-content/uploads/files/pdf_articles/Vernon_2011_1.pdf) 由 Vaughn Vernon 编写。绝对值得一读。

手动处理域事件可能很麻烦。值得庆幸的是，Spring Framework 允许我们在使用数据存储库处理聚合根 时轻松发布和处理域事件。

2。 Maven 依赖项

Spring Data 在 Ingalls 版本中引入了 @DomainEvents。它适用于任何类型的存储库。

本文提供的代码示例使用 Spring Data JPA。 将 Spring 域事件与我们的项目集成的最简单方法是使用 [Spring Boot Data JPA Starter](https://search.maven.org/search?q=g:org.springframework.boot%20AND%20a :spring-boot-starter-data-jpa):

3。手动发布事件

首先，让我们尝试手动发布域事件。我们将在下一节解释 @DomainEvents 的用法。

为满足本文的需要，我们将为域事件使用一个空的标记类—— DomainEvent。

我们将使用标准的 ApplicationEventPublisher 接口。

我们可以在两个好地方发布事件：服务层或直接在聚合内。

3.1。服务层

我们可以在服务方法中调用存储库 save 方法后简单地发布事件。

如果服务方法是事务的一部分，并且我们在使用 @TransactionalEventListener 注释的侦听器内处理事件，则只有在事务成功提交后才会处理事件。

因此，当事务回滚且聚合未更新时，不存在处理“假”事件的风险：

3.2。总计的

我们也可以直接从聚合中发布事件。

通过这种方式，我们在类中管理领域事件的创建，这感觉更自然：

@实体

    // ...
    无效域操作（）{
        // 一些业务逻辑
        如果（事件发布者！= null）{
            eventPublisher.publishEvent(new DomainEvent());
        }
    }
}

正如我们所见，该事件根本没有发布。在聚合中包含依赖关系可能不是一个好主意。在此示例中，ApplicationEventPublisher 不会由 Spring Data 自动初始化。

聚合是通过调用默认构造函数来构造的。为了使其行为符合我们的预期，我们需要手动重新创建实体（例如，使用自定义工厂或方面编程）。

此外，我们应该避免在聚合方法完成后立即发布事件。至少，除非我们 100% 确定此方法是交易的一部分。否则，当更改尚未持久化时，我们可能会发布“虚假”事件。这可能会导致系统不一致。

如果我们想避免这种情况，我们必须记住始终在事务中调用聚合方法。不幸的是，通过这种方式，我们将我们的设计与持久性技术紧密结合在一起。我们需要记住，我们并不总是使用事务系统。

因此，让我们的聚合简单地管理领域事件的集合并在即将持久化时返回它们通常是一个更好的主意。

在下一节中，我们将解释如何使用 @DomainEvents 和 @AfterDomainEvents 注释使域事件发布更易于管理。

  1. 使用 @DomainEvents 发布事件

由于 Spring Data Ingalls 发布火车，我们可以使用 @DomainEvents 注释自动发布域事件。

每当使用正确的存储库保存实体时，Spring Data 都会自动调用带有 @DomainEvents 注释的方法。

6。实施注意事项

虽然一开始使用 @DomainEvents 功能看起来是个好主意，但我们需要注意一些陷阱。

6.1。未发布的活动

在使用 JPA 时，当我们想要持久化更改时，我们不一定要调用 save 方法。

如果我们的代码是事务的一部分（例如，使用 @Transactional 注释）并对现有实体进行更改，那么我们通常只需让事务提交而不显式调用存储库上的 save 方法。因此，即使我们的聚合产生了新的领域事件，它们也永远不会被发布。

我们还需要记住 @DomainEvents 功能仅在使用 Spring Data 存储库时才有效。这可能是一个重要的设计因素。

6.2。丢失的事件

如果在事件发布期间发生异常，监听器将永远不会收到通知。

即使我们能以某种方式保证事件监听器的通知，目前也没有背压让发布者知道出了什么问题。如果事件侦听器被异常中断，则该事件将保持未使用状态，并且永远不会再次发布。

Spring 开发团队知道这个设计缺陷。一位主要开发人员甚至提出了一个可能的解决此问题的方法。

6.3。本地环境

领域事件使用简单的 ApplicationEventPublisher 接口发布。

默认情况下，当使用 ApplicationEventPublisher 时，事件在同一个线程中发布和使用。一切都发生在同一个容器中。

通常，我们希望通过某种消息代理发送事件，以便其他分布式客户端/系统得到通知。在这种情况下，我们需要手动将事件转发到消息代理。

也可以使用 Spring Integration 或第三方解决方案，例如 Apache Camel .html）。

7。小结

在本文中，我们学习了如何使用 @DomainEvents 注释来管理聚合域事件。

这种方法可以大大简化事件基础设施，因此我们可以只关注域逻辑。我们只需要知道没有灵丹妙药，Spring 处理领域事件的方式也不例外。

所有示例的完整源代码都可在 [在 GitHub 上] (https://github.com/hunterzhang86/tutorials/tree/master/persistence-modules/spring-data-jpa-annotations) 获得。
