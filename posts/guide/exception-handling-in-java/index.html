Java中的异常处理

1。概述 --------------- 在本教程中，我们将介绍 Java 中异常处理的基础知识及其一些陷阱。 2。第一原则 ----------------------- ### 2.1。它是什么？ 为了更好地理解异常和异常处理，让我们做一个真实的比较。想象一下，我们在网上订购了一个产品，但是在途中，交付失败了。一家好的公司可以处理这个问题并优雅地重新安排我们的包裹，以便它仍然准时到达。同样，在 Java 中，代码在执行我们的指令时可能会遇到错误。良好的异常处理可以处理错误并优雅地重新路由程序，从而为用户提供积极的体验*。* ### 2.2。为什么使用它？ 我们通常在理想化的环境中编写代码：文件系统总是包含我们的文件，网络是健康的，JVM 总是有足够的内存。有时我们称之为“幸福之路”。但是，在生产环境中，文件系统可能会损坏、网络崩溃以及 JVM 内存不足。我们代码的健康取决于它如何处理“不愉快的路径”。我们必须处理这些情况，因为它们会对应用程序的流程产生负面影响并形成异常：java public static List<Player> getPlayers() throws IOException { Path path = Paths.get("players.dat") ; List<String> 播放器 = 我们需要确保我们的代码有一个计划，以应对出现问题。还要注意这里对异常的另一个好处，那就是堆栈跟踪本身。由于这个堆栈跟踪，我们通常可以查明有问题的代码，而无需附加调试器。 **3。 Exception Hierarchy** -------------------------- 归根结底，*exceptions* 只是 Java 对象，它们都从 *Throwable* 扩展而来：` ``java 指的是同一件事。我们经常可以互换使用它们。 ### 3.1。 Checked Exceptions Checked 异常是 Java 编译器要求我们处理的异常。我们必须要么以声明方式将异常抛出调用堆栈，要么我们必须自己处理它。稍后会详细介绍这两种情况。 Oracle 的文档 告诉我们在我们可以合理地期望我们的方法的调用者能够恢复时使用检查的异常。已检查异常的几个示例是 IOException 和 ServletException. ### 3.2。未经检查的异常未经检查的异常是Java编译器不要求我们处理的异常。简单来说，如果我们创建了一个扩展 RuntimeException 的异常，它将被取消选中；否则，将被检查。虽然这听起来很方便，但 Oracle 的文档 告诉我们这两个概念都有充分的理由，例如区分情境错误（选中）和使用错误（未选中）。未经检查的异常的一些示例是 NullPointerException, IllegalArgumentException, 和 SecurityException。 ### 3.3。错误 错误表示严重且通常无法恢复的情况，例如库不兼容、无限递归或内存泄漏。即使它们没有扩展RuntimeException，它们也没有被选中。在大多数情况下，我们处理、实例化或扩展 Errors 会很奇怪。通常，我们希望这些一直向上传播。几个错误示例是 StackOverflowError 和 OutOfMemoryError。 4。处理异常 -------------------------- 在 Java API 中，有很多地方会出错，其中一些在签名或 Javadoc 中标记有异常的地方：java /** * @exception FileNotFoundException ... */ public Scanner(String fileName) throws FileNotFoundException { // ... } As前面说过，当我们调用这些“有风险的”方法时，我们必须处理已检查的异常，我们可能处理未检查的异常。 Java 为我们提供了几种方法来做到这一点：### 4.1。 throws “处理”异常的最简单方法是重新抛出它：java public int getPlayerScore(String playerFile) throws FileNotFoundException { Scanner contents = new Scanner(new File(playerFile)); return Integer.parseInt(contents.nextLine()); } 因为FileNotFoundException 是一个检查异常，这是满足编译器的最简单的方法，但是这确实意味着调用我们方法的任何人现在也需要处理它！ parseInt 可以抛出一个NumberFormatException，但是因为它没有被选中，所以我们不需要处理它。 ### 4.2。 try-catch 如果我们想自己尝试处理异常，我们可以使用 try-catch 块。我们可以通过重新抛出异常来处理它：java public int getPlayerScore(String playerFile) { try { Scanner contents = new Scanner(new File(playerFile)); return Integer.parseInt(contents.nextLine()); } catch (FileNotFoundException noFile) { throw new IllegalArgumentException("找不到文件"); } } 或者通过执行恢复步骤：java public int getPlayerScore(String playerFile) { try { Scanner contents = new Scanner(new File(playerFile)); return Integer.parseInt(contents.nextLine()); } catch ( FileNotFoundException noFile ) { logger.warn("文件未找到，正在重置分数。");返回0； } } ### 4.3. finally 现在，有时我们需要执行代码，无论是否发生异常，这就是 finally 关键字的用武之地。在我们目前的示例中，有一个讨厌的错误潜伏在阴影中，即默认情况下Java不会将文件句柄返回给操作系统。当然，无论我们是否可以读取文件，我们都希望确保我们进行了适当的清理！让我们先试试这个“懒惰”的方式：java public int getPlayerScore(String playerFile) throws FileNotFoundException { Scanner contents = null;尝试 { 内容 = 新扫描仪（新文件（playerFile））； return Integer.parseInt(contents.nextLine()); } 最后 { if (contents != null) { contents.close(); } } } 这里，finally 块指示我们希望 Java 运行什么代码，而不管尝试读取文件时会发生什么。即使 FileNotFoundException 被抛出调用堆栈，Java 也会在此之前调用 finally 的内容。我们也可以同时处理异常和确保我们的资源被关闭：java public int getPlayerScore(String playerFile) { Scanner contents;尝试 { 内容 = 新扫描仪（新文件（playerFile））； return Integer.parseInt(contents.nextLine()); } catch (FileNotFoundException noFile) { logger.warn("找不到文件，正在重置分数。");返回0； } 最后 { 尝试 { if (contents != null) { contents.close(); } } catch (IOException io) { logger.error("无法关闭阅读器！", io); } } } 因为 close 也是一个“有风险”的方法，我们还需要捕获它的异常！ 这可能看起来很复杂，但我们需要每个部分来处理每个可能正确出现的潜在问题. ### 4.4。 try-with-resources 幸运的是，从 Java 7 开始，我们可以在处理扩展 AutoCloseable 的内容时简化上述语法：java public int getPlayerScore(String playerFile) { try (Scanner contents =新扫描仪（新文件（playerFile）））{返回Integer.parseInt（contents.nextLine（））； } catch (FileNotFoundException e) { logger.warn("文件未找到，正在重置分数。");返回0； } } 当我们在 try 声明中放置 AutoClosable 的引用时，我们不需要自己关闭资源。不过，我们仍然可以使用 finally 块来进行我们想要的任何其他类型的清理。查看我们专门针对 try-with-resources 的文章以了解更多信息。 ### 4.5。多个 catch 块 有时，代码可能会抛出多个异常，我们可以分别处理多个 catch 块：java public int getPlayerScore(String playerFile) { try (Scanner contents = new Scanner(new File(playerFile))) { return Integer.parseInt(contents.nextLine()); } catch (IOException e) { logger.warn("播放器文件无法加载！", e);返回0； } catch (NumberFormatException e) { logger.warn("播放器文件已损坏！", e);返回0； } } 多个捕获让我们有机会在需要时以不同方式处理每个异常。还要注意，我们没有捕获 FileNotFoundException ，那是因为它 extends IOException 。因为我们正在捕获 IOException，Java 将认为它的任何子类也已处理。但是，假设我们需要将 FileNotFoundException 与更一般的 IOException 区别对待： java public int getPlayerScore(String playerFile) { try (Scanner contents = new Scanner(new File(playerFile)) ) { return Integer.parseInt(contents.nextLine()); } catch (FileNotFoundException e) { logger.warn("找不到播放器文件！", e);返回0； } catch (IOException e) { logger.warn("播放器文件无法加载！", e);返回0； } catch (NumberFormatException e) { logger.warn("播放器文件已损坏！", e);返回0； } } ``` Java 允许我们单独处理子类异常，**记得将它们放在捕获列表中更高的位置。** ### **4.6。 Union *catch* Blocks** 当我们知道我们处理错误的方式将是相同的时，Java 7 引入了在同一个块中捕获多个异常的能力：```java public int getPlayerScore(String playerFile ) { try (Scanner contents = new Scanner(new File(playerFile))) { return Integer.parseInt(contents.nextLine()); } catch (IOException | NumberFormatException e) { logger.warn("加载分数失败！", e);返回0； } } ``` **5。 Throwing Exceptions** -------------------------- 如果我们不想自己处理异常或者我们想为其他人生成我们的异常要处理，那么我们需要熟悉 *throw* 关键字。假设我们有以下我们自己创建的检查异常：```java public class TimeoutException extends Exception { public TimeoutException(String message) { super(message); } } ``` 我们有一个可能需要很长时间才能完成的方法：```java public List<Player> loadAllPlayers(String playerFile) { // ... 可能很长的操作 } ## # 5.1。抛出检查异常 就像从方法返回一样，我们可以在任何时候抛出。当然，当我们试图指出出现问题时，我们应该抛出：java public List<Player> loadAllPlayers(String playerFile) throws TimeoutException { while ( !tooLong ) { // ... 潜在的长时间操作} throw new TimeoutException("此操作耗时过长"); } 因为检查了 TimeoutException，我们还必须在签名中使用 throws 关键字，以便我们方法的调用者知道要处理它。 ### 5.2。 抛出未经检查的异常如果我们想做一些事情，比如验证输入，我们可以使用未经检查的异常：java public List<Player> loadAllPlayers(String playerFile) throws TimeoutException { if( !isFilenameValid(playersFile)) { throw new IllegalArgumentException("文件名无效！"); } // ... } 因为 IllegalArgumentException 未选中，所以我们不必标记方法，尽管我们很欢迎。无论如何，有些人将该方法标记为一种文档形式。 ### 5.3。包装和重新抛出 我们也可以选择重新抛出我们捕获的异常：java public List<Player> loadAllPlayers(String playerFile) throws IOException { try { // ... } catch (IOException io) {抛出 io; } } 或者做一个换行并重新抛出：java public List<Player> loadAllPlayers(String playerFile) throws PlayerLoadException { try { // ... } catch (IOException io) { throw new PlayerLoadException(io) ; } } 这可以很好地将许多不同的异常合并为一个。 ### 5.4。重新抛出 Throwable 或 Exception 现在是特殊情况。如果给定代码块可能引发的唯一可能异常是 unchecked 异常，那么我们可以捕获并重新抛出 Throwable 或 Exception 而无需将它们添加到我们的方法签名中：java public List<Player> loadAllPlayers(String playerFile) { try { throw new NullPointerException(); } catch (Throwable t) { throw t; } } 虽然很简单，但上面的代码不能抛出检查异常，因此，即使我们重新抛出检查异常，我们也不必用 throws 子句标记签名。 这对于代理类和方法很方便。 可以在 此处 找到更多相关信息。 ### 5.5。 继承 当我们用 throws 关键字标记方法时，它会影响子类如何覆盖我们的方法。在我们的方法抛出检查异常的情况下：java public class Exceptions { public List<Player> loadAllPlayers(String playerFile) throws TimeoutException { // ... } } risky" 签名：``java public class FewerExceptions extends Exceptions { @Override public List<Player> loadAllPlayers(String playerFile) { // 被覆盖 } } 但不是“moreriskier”签名：java public class MoreExceptions extends Exceptions { @Override public List<Player> loadAllPlayers(String playerFile) throws MyCheckedException { // overridden } } 这是因为合约在编译时由引用类型决定。如果我创建 MoreExceptions 的实例并将其保存到 Exceptions：java Exceptions exceptions = new MoreExceptions(); exceptions.loadAllPlayers("文件"); 然后 JVM 只会告诉我 catch TimeoutException ，这是错误的，因为我说过 MoreExceptions#loadAllPlayers 会引发不同的异常。 简而言之，子类可以比其超类抛出更少的检查异常，但不能更多。 6。反模式 -------------------- ### 6.1。吞下异常 现在，我们可以通过另一种方式满足编译器的要求：java public int getPlayerScore(String playerFile) { try { // ... } catch (Exception e) {} // <= = 捕获并吞下返回 0； } 以上称为 吞入异常 。大多数时候，这样做对我们来说有点意思，因为它没有解决问题并且它也使其他代码无法解决问题。有时，我们确信不会发生已检查异常。 在这些情况下，我们至少应该添加一条注释，说明我们故意吃掉了异常：java public int getPlayerScore(String playerFile) { try { // ... } catch (IOException e) { // 这永远不会发生 } } 我们可以“吞下”异常的另一种方法是简单地将异常打印到错误流中： java public int getPlayerScore(String playerFile) { try { // ... } catch (Exception e) { e.printStackTrace(); } 返回 0; }```我们至少通过在某处将错误写出来以供以后诊断，从而稍微改善了我们的情况。不过，我们最好使用记录器：```java public int getPlayerScore(String playerFile) { try { // ... } catch (IOException e) { logger.error("Couldn't加载分数", e);返回0； } } ``` 虽然以这种方式处理异常对我们来说非常方便，但我们需要确保我们不会吞下代码调用者可以用来解决问题的重要信息。最后，当我们抛出一个新异常时，我们可以通过不将其作为原因来无意中吞下一个异常：```java public int getPlayerScore(String playerFile) { try { // ... } catch (IOException e) {抛出新的 PlayerScoreException(); } } ``` 在这里，我们为提醒调用者一个错误而拍拍自己的后背，但是**我们没有将 *IOException* 作为原因。** 因此，我们丢失了调用者的重要信息或操作员可以用来诊断问题。我们最好这样做：```java public int getPlayerScore(String playerFile) { try { // ... } catch (IOException e) { throw new PlayerScoreException(e); } } ``` 注意将 *IOException* 作为 *PlayerScoreException* 的 *cause* 的细微差别。 ### **6.2。在*finally*Block** 中使用*return* 吞下异常的另一种方法是从*finally* 块中*return*。这很糟糕，因为通过突然返回，JVM 将丢弃异常，即使它是由我们的代码抛出的：```java public int getPlayerScore(String playerFile) { int score = 0;尝试 { 抛出新的 IOException(); } 最后 { 返回分数； // <== IOException 被丢弃 } } ``` 根据 [Java 语言规范](https://docs.oracle.com/javase/specs/jls/se7/html/jls-14.html#jls -14.20.2): > 如果 try 块的执行由于任何其他原因突然完成 *R*，则执行 finally 块，然后有一个选择。 > > 如果 *finally* 块正常完成，则 try 语句由于原因 R 突然完成。 > > 如果 *finally* 块由于原因 S 突然完成，则 try 语句由于原因 S 突然完成（并且原因 R 被丢弃）。 ### **6.3。在 *finally* 块中使用 *throw* 与在 *finally* 块中使用 *return* 类似，在 *finally* 块中抛出的异常将优先于在 catch 块中出现的异常。这将从 *try* 块中“删除”原始异常，并且我们会丢失所有有价值的信息：```java public int getPlayerScore(String playerFile) { try { // ... } catch (IOException io) { 抛出新的 IllegalStateException(io); // <== 被 finally 吃掉 } finally { throw new OtherException(); } } ``` ### **6.4.使用 *throw* 作为 *goto*** 一些人还接受了使用 *throw* 作为 *goto* 语句的诱惑： java public void doSomething() { try { // 一堆代码 throw new我的异常（）； // 第二组代码 } catch (MyException e) { // 第三组代码 } } ``` 这很奇怪，因为代码试图将异常用于流控制而不是错误处理。 7。常见异常和错误 ----------------------------------- 下面是一些常见的异常和错误，我们不时遇到：### 7.1。 Checked Exceptions * IOException -- 此异常通常是表示网络、文件系统或数据库上的某些内容失败的一种方式。 ### 7.2。 RuntimeExceptions * ArrayIndexOutOfBoundsException -- 这个异常意味着我们试图访问一个不存在的数组索引，就像试图从一个长度为 3 的数组中获取索引 5 时一样。 * ClassCastException -- 这个异常意味着我们尝试了执行非法强制转换，例如尝试将 String 转换为 List 。我们通常可以通过在施法前执行防御性 instanceof 检查来避免它。 * IllegalArgumentException - 这个异常是我们说提供的方法或构造函数参数之一无效的通用方式。 * IllegalStateException -- 这个异常是我们说我们的内部状态，就像我们的对象的状态一样，是无效的通用方式。 * NullPointerException -- 这个异常意味着我们试图引用一个 null 对象。我们通常可以通过执行防御性 null 检查或使用 Optional. * NumberFormatException 来避免它——这个异常意味着我们试图将 String 转换为数字，但字符串包含非法字符，例如试图将“5f3”转换为数字。 ### 7.3。 Errors * StackOverflowError -- 这个异常意味着堆栈跟踪太大。这有时会发生在大规模应用程序中；然而，这通常意味着我们的代码中发生了一些无限递归。 * NoClassDefFoundError - 这个异常意味着一个类由于不在类路径上或由于静态初始化失败而无法加载。 * OutOfMemoryError -- 这个异常意味着 JVM 没有更多的内存可以分配给更多的对象。有时，这是由于内存泄漏。 8。小结 ----------------- 在本文中，我们已经了解了异常处理的基础知识以及一些好的和不好的实践示例。与往常一样，本文中的所有代码都可以在
