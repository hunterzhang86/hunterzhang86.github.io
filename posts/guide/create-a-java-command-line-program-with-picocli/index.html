使用 Picocli 创建 Java 命令行程序

 2023.2.15  2023.5.2  手册  4769  10 分钟  
  1. 简介 --------------- 在本教程中，我们将使用 picocli 库，它可以让我们轻松创建Java中的命令行程序。我们将首先创建一个 Hello World 命令。然后，我们将通过部分复制 git 命令来深入了解该库的关键特性。 2. Hello World 命令 ---------------------- 让我们从简单的事情开始：Hello World 命令！首先，我们需要添加 对 picocli 项目的依赖项: xml <dependency> <groupId>info.picocli</groupId> <artifactId>picocli</artifactId> <version>3.9.6</version> </dependency> 正如我们所见，我们将使用 库的 3.9.6 版本，虽然 4.0.0 版本正在建设中（目前在 alpha 测试中可用）。现在已经建立了依赖关系，让我们创建我们的 Hello World 命令。为此，我们将使用库中的 @Command 注释：java @Command( name = "hello", description = "Says hello" ) public class HelloWorldCommand { } 我们可以看到，注解可以带参数。我们在这里只使用其中的两个。它们的目的是为自动帮助消息提供有关当前命令和文本的信息。目前，我们对这个命令无能为力。为了让它做点什么，我们需要添加一个 main 方法来调用 方便的 CommandLine.run(Runnable, String[]) 方法。这需要两个参数：我们的命令实例，因此它必须实现 Runnable 接口，以及表示命令参数（选项、参数和子命令）的 Stringarray：java public class HelloWorldCommand 实现可运行 { public static void main(String[] args) { CommandLine.run(new HelloWorldCommand(), args); } @Override public void run() { System.out.println("Hello World!"); } } 现在，当我们运行 main 方法时，我们会看到控制台输出 "Hello World!" 当打包到 jar 时，我们可以运行我们的 Hello World 命令使用 java 命令： shell java -cp "pathToPicocliJar;pathToCommandJar" com.baeldung.picoli.helloworld.HelloWorldCommand 毫无疑问，它也会输出 "Hello World!" * 字符串到控制台。 3. 具体用例 ---------------------- 现在我们已经了解了基础知识，我们将深入研究 picocli 库。为了做到这一点，我们将部分地复制一个流行的命令：git。当然，目的不是实现 git 命令行为，而是重现 git 命令的可能性——存在哪些子命令以及哪些选项可用于特定的子命令。首先，我们必须创建一个 GitCommand 类，就像我们为 Hello World 命令所做的那样：java @Command public class GitCommand implements Runnable { public static void main(String[] args) { CommandLine.run(new GitCommand (), 参数); } @Override public void run() { System.out.println("流行的 git 命令"); } } 4. 添加子命令 --------------------- git 命令提供了很多[子命令](https://picocli.info /#_subcommands) --- add、commit、remote 等等。我们将重点关注 add 和 commit。因此，我们的目标是将这两个子命令声明为主命令。 Picocli 提供了三种方法来实现这一点。 ### 4.1。在类上使用 @Command 注释 @Command 注释提供了通过 subcommands 参数注册子命令的可能性：java @Command( subcommands = { GitAddCommand.class, GitCommitCommand.class } ) 在我们的例子中，我们添加了两个新类： GitAddCommand 和 GitCommitCommand 。两者都用 @Command 注释并实现 Runnable 。 给它们起个名字很重要，因为 picocli 将使用这些名称来识别要执行的子命令： java @Command( name = "add" ) 公共类 GitAddCommand 实现Runnable { @Override public void run() { System.out.println("添加一些文件到暂存区"); } } ``` ```java @Command( name = "commit" ) public class GitCommitCommand implements Runnable { @Override public void run() { System.out.println("在暂存区提交文件，有多精彩？ ");因此，如果我们以 *add* 作为参数运行我们的主命令，控制台将输出 *"Adding some files to the staging area"*。 ### 4.2。在方法上使用 *@Command* 注释 声明子命令的另一种方法是在 *GitCommand* 类中**创建 *@Command* -annotated 方法来表示这些命令：```java @Command(name = "add ") public void addCommand() { System.out.println("添加一些文件到暂存区"); } @Command(name = "commit") public void commitCommand() { System.out.println("在暂存区提交文件，有多棒？"); } ``` 这样，我们可以直接在方法中实现我们的业务逻辑，而不用创建单独的类来处理它。 ### 4.3。以编程方式添加子命令 **最后，*picocli* 为我们提供了以编程方式注册子命令的可能性。** 这有点棘手，因为我们必须创建一个 *CommandLine* 对象来包装我们的命令，然后将子命令添加到它：` java CommandLine commandLine = new CommandLine(new GitCommand()); commandLine.addSubcommand("add", new GitAddCommand()); commandLine.addSubcommand("commit", new GitCommitCommand()); 在那之后，我们仍然需要运行我们的命令，但是**我们不能再使用 *CommandLine.run()* 方法**了。现在，我们必须在新创建的 C*ommandLine* 对象上调用 *parseWithHandler()* 方法：java commandLine.parseWithHandler(new RunLast(), args); 我们应该注意*RunLast* 类的使用，它告诉*picocli* 运行最具体的子命令。 *picocli* 提供了另外两个命令处理程序：*RunFirst* 和 *RunAll*。前者运行最顶层的命令，而后者运行所有命令。当使用便捷方法 *CommandLine.run()* 时，默认使用 *RunLast* 处理程序。 5. 使用 *@Option* 注解管理选项 ---------------------------------------------------- ------------ ### 5.1。无参数选项 现在让我们看看如何在命令中添加一些选项。事实上，我们想告诉我们的 *add* 命令它应该添加所有修改过的文件。为此，**我们将在我们的 *GitAddCommand* 类中添加一个带有 [*@Option*](https://picocli.info/#_options) 注释** 注释的字段：java @Option (names = {"-A", "--all"}) private boolean allFiles; @Override public void run() { if (allFiles) { System.out.println("将所有文件添加到暂存区"); } else { System.out.println("添加一些文件到暂存区"); } } 正如我们所见，注解带有一个 *names* 参数，它给出了选项的不同名称。因此，使用 *-A* 或 *--all* 调用 *add* 命令会将 *allFiles* 字段设置为 *true* 。因此，如果我们使用该选项运行命令，控制台将显示 *"Adding all files to the staging area"*。 ### 5.2。带参数的选项正如我们刚刚看到的，对于不带参数的选项，它们的存在或不存在总是被评估为一个 *boolean* 值。但是，可以注册带有参数的选项。 **我们可以简单地通过将我们的字段声明为不同类型来做到这一点。** 让我们在我们的 *commit* 命令中添加一个 *message* 选项：java @Option(names = {"-m", "--message"}) 私有字符串消息； @Override public void run() { System.out.println("在暂存区提交文件，有多棒？"); if (message != null) { System.out.println("提交信息是" + message); } } 不出所料，当给定 *message* 选项时，该命令将在控制台上显示提交消息。在本文后面，我们将介绍库处理哪些类型以及如何处理其他类型。 ### 5.3。带有多个参数的选项 但是现在，如果我们希望我们的命令接收多个消息，就像使用真正的 [*git commit*](https://git-scm.com/docs/git-commit) 命令所做的那样，该怎么办？不用担心，**让我们将字段设置为 *array* 或 *Collection***，我们已经完成了：java @Option(names = {"-m", "--message "}) 私有字符串 [] 消息； @Override public void run() { System.out.println("在暂存区提交文件，有多棒？"); if (messages != null) { System.out.println("提交信息是"); for（字符串消息：消息）{ System.out.println（消息）； } } 现在，我们可以多次使用 *message* 选项：shell commit -m "My commit is great" -m "My commit is beautiful" 但是，我们可能还想要只给出一次选项，并通过正则表达式分隔符分隔不同的参数。因此，我们可以使用 *@Option* 注释的 *split* 参数：java @Option(names = {"-m", "--message"}, split = ",") private String 消息； 现在，我们可以通过 *-m "My commit is great","My commit is beautiful"* 来达到与上面相同的结果。 ### 5.4。必需的选项 **有时，我们可能有一个必需的选项。 *required* 参数，默认为 *false*，允许我们这样做：**java @Option(names = {"-m", "--message"}, required = true) private String [] 消息； 现在不指定 *message* 选项就不可能调用 *commit* 命令。如果我们尝试这样做，*picocli* 将打印一个错误：shell Missing required option '--message=' 用法：git commit -m= [-m=] ... -m, --message= 6. 管理位置参数 ----------------------------- ---- ### 6.1。捕获位置参数 现在，让我们关注我们的 *add* 命令，因为它还不是很强大。我们只能决定添加所有文件，但如果我们想添加特定文件怎么办？我们可以使用另一个选项来做到这一点，但这里更好的选择是使用位置参数。实际上，**位置参数旨在捕获占据特定位置的命令参数，既不是子命令也不是选项。** 在我们的示例中，这将使我们能够执行以下操作：shell add file1 file2 为了捕获位置参数，**我们将使用 [*@Parameters*](https://picocli.info/#_positional_parameters) 注释**：java @Parameters private List files ; @Override public void run() { if (allFiles) { System.out.println("将所有文件添加到暂存区"); } if (files != null) { files.forEach(path -> System.out.println("Adding " + path + " to the staging area")); } } 现在，我们之前的命令将打印：shell 将file1 添加到暂存区 将file2 添加到暂存区### 6.2.捕获位置参数的子集 由于注释的 *index* 参数，可以更细粒度地捕获要捕获的位置参数。该索引从零开始。因此，如果我们定义：java @Parameters(index="2..") 这将捕获不匹配选项或子命令的参数，从第三个到结尾。索引可以是范围或单个数字，表示单个位置。 7. 关于类型转换 ------------------- 正如我们在本教程前面所看到的，*picocli * 自己处理一些类型转换。例如，它将多个值映射到 *arrays* 或 *Collections* ，但它也可以将参数映射到特定类型，例如当我们将 *Path* 类用于 *add* 命令时。 **事实上，*picocli* 带有 [一堆预先处理的类型](https://picocli.info/#_built_in_types)。这意味着我们可以直接使用这些类型，而不必考虑自己转换它们。** 但是，我们可能需要将命令参数映射到已经处理的类型之外的类型。对我们来说幸运的是，**这要归功于 [*ITypeConverter*](https://picocli.info/#_custom_type_converters) 接口和 *CommandLine#registerConverter* 方法，它将类型与转换器相关联**。假设我们想将 *config* 子命令添加到 *git* 命令中，但我们不希望用户更改不存在的配置元素。因此，我们决定将这些元素映射到一个枚举：java public enum ConfigElement { USERNAME("user.name"), EMAIL("user.email");私有最终字符串值； ConfigElement(String value) { this.value = value; } public String value() { 返回值； } public static ConfigElement from(String value) { return Arrays.stream(values()) .filter(element -> element.value.equals(value)) .findFirst() .orElseThrow(() -> new IllegalArgumentException("The参数 " + value + " 不匹配任何 ConfigElement")); } } 另外，在我们新创建的 *GitConfigCommand* 类中，让我们添加两个位置参数： ``java @Parameters(index = "0") private ConfigElement element; @Parameters(index = "1") 私有字符串值； @Override public void run() { System.out.println("设置 " + element.value() + " 到 " + value); } 这样，我们确保用户将无法更改不存在的配置元素。最后，我们必须注册我们的转换器。美妙的是，如果使用 Java 8 或更高版本，我们甚至不必创建实现 ITypeConverter 接口的类。 我们可以将 lambda 或方法引用传递给 registerConverter() 方法： java CommandLine commandLine = new CommandLine(new GitCommand()); commandLine.registerConverter(ConfigElement.class, ConfigElement::from); commandLine.parseWithHandler(new RunLast(), args); 这发生在 GitCommand main() 方法中。请注意，我们必须放弃方便的 CommandLine.run() 方法。当与未处理的配置元素一起使用时，该命令将显示帮助消息以及一条信息，告诉我们无法将参数转换为 ConfigElement：shell Invalid value for positional parameter at index 0 (<element>): 无法将 'user.phone' 转换为 ConfigElement (java.lang.IllegalArgumentException: 参数 user.phone 不匹配任何 ConfigElement) 用法: git config <element> <value> <element> <value > 8. 与 Spring Boot 集成 ------------------- 最后，让我们看看如何 Springify 所有这些！事实上，我们可能在 Spring Boot 环境中工作，并希望在我们的命令行程序中从中受益。为此，我们必须创建一个 SpringBootApplication 实现 CommandLineRunner 接口：java @SpringBootApplication public class Application implements CommandLineRunner { public static void main(String[] args) { SpringApplication.运行（应用程序。类，参数）； } @Override public void run(String... args) { } } 另外，让我们用 Spring @Component 注释注释我们所有的命令和子命令，并在我们的 Application 中自动装配所有这些： java 私有 GitCommand gitCommand;私有 GitAddCommand addCommand;私有 GitCommitCommand 提交命令；私有 GitConfigCommand 配置命令；公共应用程序（GitCommand gitCommand，GitAddCommand addCommand，GitCommitCommand commitCommand，GitConfigCommand configCommand）{ this.gitCommand = gitCommand; this.addCommand = addCommand; this.commitCommand = commitCommand; this.configCommand = 配置命令； } 请注意，我们必须自动装配每个子命令。不幸的是，这是因为目前 picocli 在以声明方式（带有注释）声明时还不能从 Spring 上下文中检索子命令。因此，我们必须自己进行接线，以编程方式：java @Override public void run(String... args) { CommandLine commandLine = new CommandLine(gitCommand); commandLine.addSubcommand("add", addCommand); commandLine.addSubcommand("commit", commitCommand); commandLine.addSubcommand("config", configCommand); commandLine.parseWithHandler(new CommandLine.RunLast(), args); } 现在，我们的命令行程序就像使用 Spring 组件的魅力一样工作。因此，我们可以创建一些服务类并在我们的命令中使用它们，让 Spring 来处理依赖注入。 9. 结论 ------------- 在本文中，我们看到了 picocli 库的一些关键特性。我们已经学习了如何创建一个新命令并向其添加一些子命令。我们已经看到了许多处理选项和位置参数的方法。此外，我们还学习了如何实现我们自己的类型转换器以使我们的命令具有强类型。最后，我们看到了如何将 Spring Boot 引入我们的命令。当然，关于它还有更多的发现。该库提供 完整文档。至于本文的完整代码，可以在【我们的 GitHub】（https://github.com/hunterzhang86/tutorials/tree/master/libraries-2）上找到。 Java 底层 ### 通过 Learn Spring 课程开始使用 Spring 5 和 Spring Boot 2： >> 查看课程** 通用页脚横幅！使用 SPRING 构建您的 API - 书籍封面 ！使用 SPRING 构建您的 API - 图标 学习使用 Spring 构建 API **？[下载电子书](/休息API弹簧指南）
