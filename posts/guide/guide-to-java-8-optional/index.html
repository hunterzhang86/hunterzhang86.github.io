JDK 9 *Optional* API ------------------------ Java 9 的发布为 *Optional 添加了更多新方法* API：* *or()* 方法，用于提供创建替代 *Optional* * *ifPresentOrElse()* 方法，如果 *Optional* 存在则允许执行操作，如果不存在则允许执行另一个操作 * *stream() * 将 *Optional* 转换为 *Stream* 的方法 这是 [进一步阅读](/java-9-optional) 的完整文章。 15. *Optional* s 的误用 -------------------------- 最后，让我们看一下使用 *Optional* 的诱人但危险的方法s：将*可选*参数传递给方法。想象一下，我们有一个 *Person* 列表，我们想要一种方法来在该列表中搜索具有给定名称的人。此外，如果指定了该方法，我们希望该方法能够匹配至少具有一定年龄的条目。由于这个参数是可选的，我们使用了这个方法：java public static List search(List people, String name, Optional age) { // Null 检查人员和名称返回people.stream() .filter(p -> p.getName().equals(name)) .filter(p -> p.getAge().get() >= age.orElse(0)) .collect(Collectors .toList()); } 然后我们发布我们的方法，另一个开发者尝试使用它：java someObject.search(people, "Peter", null); 现在开发人员执行它的代码并得到一个 *NullPointerException。* **我们必须对我们的可选参数进行空检查，这违背了我们想要避免这种情况的最初目的。** 这里有一些可能性我们本可以做得更好：java public static List search(List people, String name, Integer age) { // Null 检查人员和名称 final Integer ageFilter = age !=无效的 ？年龄：0； return people.stream() .filter(p -> p.getName().equals(name)) .filter(p -> p.getAge().get() >= ageFilter) .collect(Collectors.toList() ); } 在那里，参数仍然是可选的，但我们只在一次检查中处理它。另一种可能性是**创建两个重载方法**：java public static List search(List people, String name) { return doSearch(people, name, 0); } public static List search(List people, String name, int age) { return doSearch(people, name, age); } private static List doSearch(List people, String name, int age) { // 对人员和姓名进行空检查 return people.stream() .filter(p -> p.getName().equals( name)) .filter(p -> p.getAge().get().intValue() >= age) .collect(Collectors.toList()); } ``` 这样我们就提供了一个清晰的 API，其中有两种方法做不同的事情（尽管它们共享实现）。因此，有一些解决方案可以避免使用 Optional 作为方法参数。Java 在释放 Optional 时的意图是将其用作返回类型 ，从而表明方法可以返回空值。事实上，使用 Optional 作为方法参数的做法甚至 被一些代码检查员反对。 16. Optional 和序列化 -------------------------------- 如上所述，Optional 意味着用作返回类型。不建议尝试将其用作字段类型。此外，在可序列化的类中使用 Optional 将导致 NotSerializableException。 我们的文章 Java Optional as Return Type 进一步解决了序列化问题。而且，在 使用 Optional 与 Jackson 中，我们解释了当 Optional 字段被序列化时会发生什么，以及一些解决方法来实现所需的结果。 17。小结 ------------------ 在本文中，我们介绍了 Java 8 Optional 类的大部分重要特性。我们简要探讨了为什么我们会选择使用 Optional 而不是显式的 null 检查和输入验证的一些原因。我们还学习了如何使用 get() 、 orElse() 和 orElseGet() 方法获取 Optional 的值，或者默认值（如果为空）（并看到 the important difference between the最后两个)。然后我们看到了如何使用 map()、flatMap() 和 filter() 转换或过滤我们的 Optional 。我们讨论了 fluent API Optional 提供了什么，因为它允许我们轻松地链接不同的方法。最后，我们看到了为什么使用 Optionals 作为方法参数是一个坏主意以及如何避免它。文章中所有示例的源代码都可在 [在 GitHub 上] (https://github.com/hunterzhang86/tutorials/tree/master/core-java-modules/core-java-11-2) Java 底部### 通过 Learn Spring 课程开始使用 Spring 5 和 Spring Boot 2： >> 查看课程** 通用页脚横幅 学习使用 Spring 构建 API **？[下载电子书](/rest- api-spring-guide)
