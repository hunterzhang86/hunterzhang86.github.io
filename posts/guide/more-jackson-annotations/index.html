更多杰克逊注解

 2023.4.13  2023.5.3  手册  3510  8 分钟  

1。概述 ---------------- 这篇文章涵盖了一些在上一篇文章 A Guide to Jackson Annotations 中没有涉及到的额外注解——我们将经历其中的七个。 2。 @JsonIdentityReference ------------------- @JsonIdentityReference 用于自定义对以下对象的引用将被序列化为对象身份而不是完整的 POJO。它与 @JsonIdentityInfo 协作以强制在每个序列化中使用对象身份，这与不存在 @JsonIdentityReference 时的所有但第一次不同。在处理对象之间的循环依赖关系时，这对注释最有用。有关详细信息，请参阅 Jackson -- 双向关系 文章的第 4 节。为了演示 @JsonIdentityReference 的使用，我们将定义两个不同的 bean 类，没有和有这个注解。没有 @JsonIdentityReference 的 bean：java @JsonIdentityInfo(generator = ObjectIdGenerators.PropertyGenerator.class, property = "id") public class BeanWithoutIdentityReference { private int id;私有字符串名称； // 构造函数、getter 和 setter } 对于使用 @JsonIdentityReference 的 bean，我们选择 id 属性作为对象标识： java @JsonIdentityInfo(generator = ObjectIdGenerators.PropertyGenerator.class, property = "id") @JsonIdentityReference(alwaysAsId = true) 公共类 BeanWithIdentityReference { private int id;私有字符串名称； // 构造函数、getter 和 setter } ``` 在第一种情况下，*@JsonIdentityReference* 不存在，该 bean 被序列化并带有其属性的完整详细信息：```java BeanWithoutIdentityReference bean = new BeanWithoutIdentityReference(1, "Bean没有身份参考注释");字符串 jsonString = mapper.writeValueAsString(bean); ``` 上面序列化的输出： ```java { "id": 1, "name": "Bean without Identity Reference Annotation" } ``` 当使用*@JsonIdentityReference*时，bean被序列化为一个简单的身份代替：```java BeanWithIdentityReference bean = new BeanWithIdentityReference(1, "Bean With Identity Reference Annotation");字符串 jsonString = mapper.writeValueAsString(bean); assertEquals("1", jsonString); ``` **3。 *@JsonAppend*** -------------------- *@JsonAppend* 注释用于在对象被添加时向对象添加常规属性之外的虚拟属性序列化。当我们想要将补充信息直接添加到 JSON 字符串中而不是更改类定义时，这是必要的。例如，将 bean 的 *version* 元数据插入相应的 JSON 文档可能比为其提供附加属性更方便。假设我们有一个没有 *@JsonAppend* 的 bean，如下所示：```java public class BeanWithoutAppend { private int id;私有字符串名称； // 构造函数、getter 和 setter } ``` 测试将确认在没有 *@JsonAppend* 注释的情况下，序列化输出不包含关于补充 *version* 属性的信息，尽管我们尝试添加到 *ObjectWriter* 对象：```java BeanWithoutAppend bean = new BeanWithoutAppend(2, "Bean without Append Annotation"); ObjectWriter writer = mapper.writerFor(BeanWithoutAppend.class).withAttribute("version", "1.0");字符串 jsonString = writer.writeValueAsString(bean); ``` 序列化输出： ```java { "id": 2, "name": "Bean without Append Annotation" } ``` 现在，假设我们有一个用 *@JsonAppend* 注释的 bean： `java @JsonAppend(attrs = { @JsonAppend.Attr(value = "version") }) public class BeanWithAppend { private int id;私有字符串名称； // 构造函数、getter 和 setter } 与上一个类似的测试将验证应用 *@JsonAppend* 注解时，在序列化后包含补充属性： ``java BeanWithAppend bean = new BeanWithAppend( 2、《带有附加注解的Bean》）； ObjectWriter writer = mapper.writerFor(BeanWithAppend.class).withAttribute("version", "1.0");字符串 jsonString = writer.writeValueAsString(bean); 该序列化的输出显示 version 属性已添加：java { "id": 2, "name": "Bean With Append Annotation", "version": "1.0" } 4。 @JsonNaming -------------------- @JsonNaming 注解用于选择序列化中属性的命名策略，覆盖默认值。使用 value 元素，我们可以指定任何策略，包括自定义策略。除了默认的 LOWER_CAMEL_CASE（例如 lowerCamelCase），Jackson 库还为我们提供了四种其他内置属性命名策略以方便使用： * KEBAB_CASE ：名称元素由连字符分隔，例如烤肉串盒。 * LOWER_CASE ：所有字母都是小写，没有分隔符，例如小写。 * SNAKE_CASE : 所有字母都是小写，下划线作为名称元素之间的分隔符，例如snake_case。 * UPPER_CAMEL_CASE ：所有名称元素，包括第一个，以大写字母开头，后跟小写字母，并且没有分隔符，例如大驼峰式。此示例将说明使用蛇案例名称序列化属性的方法，其中名为 beanName 的属性被序列化为 bean_name。 给定一个 bean 定义：java @JsonNaming(PropertyNamingStrategies.SnakeCaseStrategy.class) public class NamingBean {私人int id；私有字符串 beanName； // 构造函数、getter 和 setter } 下面的测试表明指定的命名规则按要求工作： java NamingBean bean = new NamingBean(3, "Naming Bean");字符串 jsonString = mapper.writeValueAsString(bean); assertThat(jsonString, containsString("bean_name")); jsonString 变量包含以下数据：java { "id": 3, "bean_name": "Naming Bean" } 5。 @JsonPropertyDescription --------------------------------- Jackson 库能够为 Java 创建 JSON 模式在名为 JSON Schema 的单独模块的帮助下输入类型。当我们想要在序列化 Java 对象时指定预期输出或在反序列化之前验证 JSON 文档时，该模式很有用。 @JsonPropertyDescription 注释允许通过提供 description 字段将人类可读的描述添加到创建的 JSON 模式中。本节使用下面声明的 bean 来演示 @JsonPropertyDescription 的功能：java public class PropertyDescriptionBean { private int id; @JsonPropertyDescription("这是对name属性的描述") private String name; // getter 和 setter } 通过添加 description 字段生成 JSON 模式的方法如下所示： java SchemaFactoryWrapper wrapper = new SchemaFactoryWrapper(); mapper.acceptJsonFormatVisitor(PropertyDescriptionBean.class, wrapper); JsonSchema jsonSchema = wrapper.finalSchema();字符串 jsonString = mapper.writeValueAsString(jsonSchema); assertThat(jsonString, containsString("这是对name属性的描述")); 可以看到，JSON schema 生成成功：java { "type": "object", "id": "urn:jsonschema:com:baeldung:jackson:annotation:extra:PropertyDescriptionBean ", "properties": { "name": { "type": "string", "description": "这是对 name 属性的描述" }, "id": { "type": "integer" } } } 6。 @JsonPOJOBuilder ------------- @JsonPOJOBuilder 注解用于配置构建器类以自定义 JSON 的反序列化当命名约定与默认值不同时恢复 POJO 的文档。假设我们需要反序列化以下 JSON 字符串：java { "id": 5, "name": "POJO Builder Bean" } 该 JSON 源将用于创建 POJOBuilderBean 的实例: java @JsonDeserialize(builder = BeanBuilder.class) public class POJOBuilderBean { private int identity;私有字符串 beanName； // 构造函数、getter 和 setter } bean 的属性名称与 JSON 字符串中的字段名称不同。这就是 @JsonPOJOBuilder 来救援的地方。 @JsonPOJOBuilder 注释伴随着两个属性： * buildMethodName ：在将 JSON 字段绑定到该 bean 的属性后，用于实例化预期 bean 的无参数方法的名称。默认名称是 build。 * withPrefix : 用于自动检测 JSON 和 bean 属性之间匹配的名称前缀。默认前缀是 with。这个例子使用下面的 BeanBuilder 类，它在 POJOBuilderBean 上使用： java @JsonPOJOBuilder(buildMethodName = "createBean", withPrefix = "construct") public class BeanBuilder { private int idValue;私有字符串名称值；公共 BeanBuilder 构造 ID(int id) { idValue = id;返回这个； } 公共 BeanBuilder 构造名称（字符串名称） { nameValue = name;返回这个； } 公共 POJOBuilderBean createBean() { return new POJOBuilderBean(idValue, nameValue); } } 在上面的代码中，我们将 @JsonPOJOBuilder 配置为使用名为 createBean 的构建方法和匹配属性的 construct 前缀。 @JsonPOJOBuilder对bean的应用描述和测试如下：java String jsonString = "{\"id\":5,\"name\":\"POJO Builder Bean\"}" ; POJOBuilderBean bean = mapper.readValue(jsonString, POJOBuilderBean.class); assertEquals(5, bean.getIdentity()); assertEquals("POJO Builder Bean", bean.getBeanName()); 结果表明，尽管属性名称不匹配，但已成功从 JSON 源重新创建新数据对象。 7。 @JsonTypeId -------------------- @JsonTypeId 注解用于指示被注解的属性在包含多态时应该序列化为类型id类型信息，而不是作为常规属性。该多态元数据在反序列化期间用于重新创建与序列化之前相同子类型的对象，而不是声明的超类型。有关 Jackson 处理继承的更多信息，请参阅 Inheritance in Jackson 的第 2 节。假设我们有一个 bean 类定义如下：java public class TypeIdBean { private int id; @JsonTypeId 私有字符串名称； // 构造函数、getter 和 setter } 下面的测试验证 @JsonTypeId 可以正常工作： java mapper.enableDefaultTyping(DefaultTyping.NON_FINAL); TypeIdBean bean = new TypeIdBean(6, "Type Id Bean");字符串 jsonString = mapper.writeValueAsString(bean); assertThat(jsonString, containsString("Type Id Bean")); 序列化过程' 输出：java [ "Type Id Bean", { "id": 6 } ] 8. @JsonTypeIdResolver ---------------------------- @JsonTypeIdResolver 注解用于表示自定义类型标识处理程序序列化和反序列化。该处理程序负责 Java 类型和 JSON 文档中包含的类型 id 之间的转换。假设我们想在处理以下类层次结构时将类型信息嵌入到 JSON 字符串中。 AbstractBean 超类：java @JsonTypeInfo(use = JsonTypeInfo.Id.NAME, include = JsonTypeInfo.As.PROPERTY, property = "@type") @JsonTypeIdResolver(BeanIdResolver.class) public class AbstractBean { private int ID;受保护的 AbstractBean(int id) { this.id = id; } // 无参数构造函数，getter 和 setter } FirstBean 子类：java public class FirstBean extends AbstractBean { String firstName; public FirstBean(int id, String name) { super(id); setFirstName(名字); } // 无参数构造函数，getter 和 setter } LastBean 子类：java public class LastBean extends AbstractBean { String lastName; public LastBean(int id, String name) { super(id);设置姓氏（名称）； } // 无参数构造函数，getter 和 setter } 这些类的实例用于填充 BeanContainer 对象：java public class BeanContainer { private List<AbstractBean> beans; // getter 和 setter } 我们可以看到 AbstractBean 类被注解了 @JsonTypeIdResolver ，表示它使用自定义的 TypeIdResolver 来决定如何在序列化中包含子类型信息以及如何使用反过来说元数据。这是处理包含类型信息的解析器类：java public class BeanIdResolver extends TypeIdResolverBase { private JavaType superType; @Override public void init(JavaType baseType) { superType = baseType; } @Override public Id getMechanism() { return Id.NAME; } @Override public String idFromValue(Object obj) { return idFromValueAndType(obj, obj.getClass()); } @Override public String idFromValueAndType(Object obj, Class<?> subType) { String typeId = null; switch (subType.getSimpleName()) { case "FirstBean": typeId = "bean1";休息; case "LastBean": typeId = "bean2"; } 返回类型标识； } @Override public JavaType typeFromId(DatabindContext context, String id) { Class<?> subType = null; switch (id) { case "bean1": subType = FirstBean.class;休息; case "bean2": subType = LastBean.class; } 返回 context.constructSpecializedType(superType, subType); } } 两个最值得注意的方法是 idFromValueAndType 和 typeFromId，前者告诉在序列化 POJO 时包含类型信息的方式，后者使用该元数据确定重新创建对象的子类型。为了确保序列化和反序列化都能正常工作，让我们编写一个测试来验证完整的进度。首先，我们需要实例化一个 bean 容器和 bean 类，然后用 bean 实例填充该容器：java FirstBean bean1 = new FirstBean(1, "Bean 1"); LastBean bean2 = new LastBean(2, "Bean 2"); List<AbstractBean> beans = new ArrayList<>(); beans.add(bean1); beans.add(bean2); BeanContainer serializedContainer = new BeanContainer(); serializedContainer.setBeans(beans); 接下来，BeanContainer 对象被序列化，我们确认生成的字符串包含类型信息： java String jsonString = mapper.writeValueAsString(serializedContainer); assertThat(jsonString, containsString("bean1")); assertThat(jsonString, containsString("bean2")); 序列化的输出如下图所示： java { "beans": [ { "@type": "bean1", "id": 1, "firstName": "Bean 1" }, { " @type": "bean2", "id": 2, "lastName": "Bean 2" } ] } 该 JSON 结构将用于重新创建与序列化之前相同子类型的对象。下面是反序列化的实现步骤：java BeanContainer deserializedContainer = mapper.readValue(jsonString, BeanContainer.class); List<AbstractBean> beanList = deserializedContainer.getBeans(); assertThat(beanList.get(0), instanceOf(FirstBean.class)); assertThat(beanList.get(1), instanceOf(LastBean.class)); 9。小结 ----------------- 本教程详细解释了几个不太常见的 Jackson 注解。这些示例和代码片段的实现可以在 GitHub 项目 中找到。 Jackson bottom ### 通过 Learn Spring 课程开始使用 Spring 5 和 Spring Boot 2：****>> 查看课程 Jackson页脚横幅！[与杰克逊一起做 JSON - 书籍封面]（wp-content/uploads/2016/05/baeldung-jackson-post-footer-main-1.2.0.jpg）！[与杰克逊一起做 JSON - 图标] (wp-content/uploads/2016/05/baeldung-jackson-lightbox-icn-1_kraken.png) 与杰克逊一起做 JSON 下载电子书
