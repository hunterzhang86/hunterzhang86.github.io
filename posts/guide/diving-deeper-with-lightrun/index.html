与 Lightrun 一起深入探索

 2023.4.22  2023.5.10  手册  6400  13 分钟  

1。介绍 -------------------- 在我们的 上一篇文章 中，我们介绍了 Lightrun -- 一个开发人员可观察性平台. 在本文中，我们将深入研究它提供的功能，我们如何才能最好地在我们的应用程序中使用它们，以及我们可以从中获得什么。 2.快照 ---------------- 在上一篇文章中，我们简要介绍了 快照 是什么以及它们可以为我们做什么。在这里，我们将更深入地研究它们是什么，我们如何最好地使用它们，以及它们可以为我们做什么。 **快照类似于调试器断点。**我们可以在应用程序的任何代码行上注册快照。每次触发它都会自动记录完整的堆栈跟踪和每个可见变量的值。就像正常的断点一样，这将包括局部变量、方法参数和类字段，并且会在整个堆栈帧中进行。快照和调试器断点之间的主要区别在于快照是非侵入式的。它们不会导致任何东西阻塞——无论是整个应用程序还是只是正在执行的线程。它们记录当前的执行状态，并且应用程序继续运行而不会以任何方式中断应用程序。传统的阻塞断点（例如在调试器中）会导致单线程或整个应用程序在我们查看细节时停止暂停。 Lightrun 允许我们在不影响实时应用程序的情况下完成所有这些工作。 ### 2.1。放置快照 快照直接从我们的代码编辑器放置到我们的应用程序中。 在本文中，我们使用的是 IntelliJ IDEA，但也可以通过 Visual Studio Code 实现一切。我们需要确定要在应用程序中放置快照的位置。然后我们可以右键单击这行代码，从菜单中选择“Lightrun > Snapshot (Virtual Breakpoint)”：![](https://www.baeldung.com/wp-content/uploads/2022/ 09/place-snapshot.png) 这样做会打开一个对话框，允许我们指定快照的详细信息：![](https ://www.baeldung.com/wp-content/uploads/2022/09/create-snapshot.png) 默认行为这相对简单，但通常这是最有用的。 它将： * 拍摄所选行的快照。 * 没有关于何时触发快照的条件。 * 没有额外的表达式来记录快照。 * 仅在第一次触发快照时记录。 * 添加后一小时过期。这意味着它将在下一次执行这行代码时记录确切的执行状态，只要它是在下一小时内。当我们诊断问题时，这通常是最有用的设置，因为我们希望控制录制的内容，而不会在录制中产生任何噪音。我们将在本文后面看到更多关于所有这些的信息。完成此操作后，将在注册快照的代码行旁边放置一个蓝色相机图标。这说明我们的快照已经放置成功，触发时会记录：[](/wp -content/uploads/2022/09/snapshot-added.png) 当快照被触发时，我们将自动在我们的编辑器中查看详细信息。这看起来和功能几乎与 IntelliJ 断点面板完全相同，因为它旨在服务于相同的目的：![](https://www.baeldung.com/wp-content/uploads/2022/09/snapshot -triggered.png) 在这里我们可以立即看到我们执行的完整堆栈跟踪，以及我们可以访问的变量。这包括变量 this，在其中我们可以看到当前类实例的字段。我们还可以根据需要深入研究这些方法，并单击堆栈框架中的其他方法，与使用传统调试器的方式完全相同。 ### 2.2。条件快照 **在某些情况下，我们希望仅在满足某些条件时记录我们的快照。**例如，仅当当前用户是特定用户名时。 Lightrun 快照允许我们指定条件作为设置它们的一部分。这与我们调试器中条件断点的工作方式非常相似。我们的条件被指定为一个 Java 表达式，其计算结果为 true 或 false。这可以访问在触发快照时可见的任何内容，这意味着任何局部变量、参数、类字段或其他任何内容。例如，假设我们有一个带有参数 id 的方法。我们希望在调用它时记录快照，但前提是提供的 ID 是特定值。我们可以根据需要设置一个完全触发的条件：[](/wp-content /uploads/2022/09/conditional-snapshot.png）这意味着快照只会在使用我们的特定测试值调用时触发，但除非它们碰巧使用相同的值，否则任何其他实时使用的服务都将被忽略。这有助于确保我们在快照窗格中看到的正是我们想要的，而没有任何额外的噪音干扰它并使我们的诊断更加困难。 ### 2.3。附加表达式 在某些情况下，我们可能需要将附加值记录为快照的一部分。 这些可能是其他值的计算，只是为了让生活更轻松——例如，钻入嵌套值以使它们更容易浮出水面。这些也可能是调用以获取否则不会记录的值 - 例如，来自静态变量，例如 [RequestContextHolder](https://docs.spring.io/spring-framework/docs/current/javadoc -api/org/springframework/web/context/request/RequestContextHolder.html) 或 SecurityContextHolder。它们甚至可以是对我们可以看到的任何值的方法调用，并记录这些方法的结果。表达式以与条件非常相似的方式添加，通过在快照对话框中输入要记录的表达式：![](https://www.baeldung.com/wp-content/uploads/2022/09/snapshot- expression.png) 我们可以根据需要在单个快照中添加任意数量的表达式，并且在触发快照时将计算并记录所有表达式.然后这些值作为“变量”窗格的一部分出现在录制的快照中，并带有不同的图标，表示它们是手动添加的表达式，而不是自动检测到的变量：![](https://www.baeldung.com/ wp-content/uploads/2022/09/snapshot-variables.png) ### 2.4。记录多个快照 在某些情况下，我们可能希望从同一个地方记录多个快照。例如，我们可能希望通过系统运行几个略有不同的请求，并能够比较它们的快照以识别差异。 Lightrun Snapshots 默认只记录一个快照，但我们可以通过在“快照”对话框中输入最大命中数来将其配置为记录任意数量的快照：![](https://www.baeldung.com/ wp-content/uploads/2022/09/multiple-snapshot.png) 这样做会记录这个执行次数的快照，并制作它们在我们的编辑器中提供给我们：[](/wp-content/uploads/2022 /09/view-multiples-snapshots.png) 现在我们有多个快照可以使用，我们需要知道哪些是哪些。通过单击“快照”选项卡一侧的“i”图标，我们会看到有关此确切快照的信息对话框：![](https://www.baeldung.com/wp-content/uploads/2022/ 09/snapshot-info.png) 在这里我们可以看到记录快照的服务器实例和记录时间。我们现在可以根据需要记录尽可能多的快照，并确定哪些是哪些，以便我们可以更好地诊断正在发生的事情。 ### 2.5。自动过期快照 记录快照对我们的应用程序的性能成本确实很小。它们还会导致数据从我们的应用程序传输到 Lightrun 服务器，这可能会产生成本。这意味着虽然快照对于诊断问题非常有用，但我们希望确保它们不会停留超过所需时间。 Lightrun 通过自动过期快照为我们解决了这个问题，因此它们只会影响我们的应用程序以满足我们的确切需求，而不会影响更多。 默认情况下，快照将在 1 小时后自动禁用。如果我们想以最小的影响对自己进行一些集中测试，我们可以将其设置为非常短的时间。或者，我们可以将其设置为一个很长的时间段，例如，以捕捉在一夜之间、一个周末甚至更长时间内发生的任何特定问题的发生。我们可以从“创建快照”对话框的“高级”部分调整快照将处于活动状态的持续时间。这为我们提供了一个额外的到期时间选项，它允许我们以小时、分钟和秒为单位指定快照的活动时间：![](https://www.baeldung.com/wp- content/uploads/2022/09/snapshot-expiry.png) 在这段时间之后，快照将保持存在，以便记录的快照仍然存在可用。 但是，它会停止记录任何内容——即使我们还没有达到最大命中数。如果我们不更改此设置，则默认值为 1 小时。这段时间过去后，我们快照的相机图标变为红色，表示它不再处于活动状态：![](https://www.baeldung.com/wp-content/uploads/2022/09/stop-recording .png) 请注意，快照仍保留在我们的系统中，否则，记录的数据将不可用。但是，除非重新启用，否则它不会再记录任何数据。 3。日志 ----------- **Lightrun 提供的另一个功能是能够将 日志记录语句 动态添加到我们的应用程序中，而无需更改或重新启动任何内容。**日志的配置方式与快照类似，但用途不同。快照通过在触发它们时记录线程的确切状态来工作。相反，日志会将所需信息写入日志流。这意味着许多日志消息——无论是内置在应用程序中还是由 Lightrun 动态添加——将在日志流中混合在一起，并给出正在发生的事情的更大图景。我们将看到来自我们应用程序的日志消息和 Lightrun 添加的任何日志合并到同一个流中，让我们全面了解正在发生的事情。 ### 3.1。添加动态日志 使用 Lightrun 添加动态日志的方式与添加快照非常相似。 我们右键单击之前要添加日志语句的行，然后从菜单：[](/wp-content/uploads/2022/09/dynamic-log. png）然后给我们一个对话框来配置动态日志语句，然后将其添加到我们正在运行的应用程序中：![](https://www.baeldung.com/wp-content/uploads/2022/09/ create-log.png) 这使我们能够指定将输出的日志消息——其中可以包括动态表达式作为信息。我们还可以指定触发日志消息所需的条件，与快照条件的工作方式完全相同。默认情况下，这些日志消息将在 1 小时后过期，但也可以通过单击“高级”按钮来更改与快照相同的设置。日志消息也有一个日志级别，默认为 INFO，但我们可以根据需要将它们更改为 DEBUG、WARN 或 ERROR。一旦我们添加了日志语句，编辑器将在代码视图中指出这一点，以显示日志语句在哪里以及它在做什么：![](https://www.baeldung.com/wp-content/ uploads/2022/09/log-statement.png) ### 3.2.查看日志 **默认情况下，我们的动态日志消息是使用 Java Util Logging 写出的。**在这种情况下，我们可以看到它们与应用程序生成的任何其他日志消息交错，这可以提供更多信息： 也可以将日志消息发送到我们的编辑器以在本地查看。 这些可以在 Lightrun 控制台中查看，类似于我们查看快照的方式。如果我们想在不给输出日志文件添加额外噪音的情况下向系统添加日志记录，这将非常有用，尤其是当其他团队成员或其他系统正在使用这些日志时：![](https://www. baeldung.com/wp-content/uploads/2022/09/lightrun-console-logs.png) 我们可以更改日志消息的位置通过打开侧栏中的代理菜单并选择日志管道来输出：[](/wp -content/uploads/2022/09/log-piping.png) 从这里我们可以选择 App - 这意味着写入应用程序的已配置 Java Util Logging 设置，Plugin - 这意味着写入活动的 Lightrun 插件我们的编辑，或两者兼而有之。请注意，这是针对整个 Lightrun 代理而不是针对单个日志消息执行的。由于 Lightrun 代理的工作方式，Java Util Logging 配置不是应用程序的标准配置。相反，需要一些 Lightrun 代理标志 来配置 Lightrun 动态记录器在写入时的目标和输出格式到 Java Util 日志记录。 ### 3.3。记录表达式 记录简单字符串已经很有用了。但是，记录应用程序中的值要有用得多。 就像我们的快照可以包含自定义表达式一样，我们可以为日志执行此操作。当我们对日志执行此操作时，我们将表达式直接添加到日志消息中。这是通过将表达式包裹在花括号中来完成的：plaintext 搜索任务：status={status}, createdBy={createdBy} 当我们这样做时，这些表达式中的任何一个都将在 log 语句时自动扩展生成：[](/wp-content/uploads/2022/09/logging-expressions .png) 这些表达式可以是在生成日志语句时可以确定的任何内容，其方式与快照完全相同。这些表达式有时会占用大量 CPU 时间来计算。如果发生这种情况，Lightrun 可能会自动暂停特定日志，以便它们不会干扰应用程序的运行。因此，建议保持日志表达式尽可能简单。 4。 Metrics -------------- 我们可以使用 Lightrun 执行的最终操作是记录一些关于我们的应用程序的 metrics。 这给了我们查看我们应用程序的使用细节的能力——例如，某些事情发生的频率或需要多长时间。与快照和日志一样，通过右键单击相应的代码行并选择“Lightrun > Metrics”来添加指标：![](https://www.baeldung.com/wp-content /uploads/2022/09/lightrun-metrics.png) 我们可以立即看到这略有不同——我们有不同类型的指标我们可以添加： * Counter -- 记录一行代码执行次数的简单计数。 * Time Measure - 这记录了两行代码之间的时间。* 方法持续时间——这记录了进入和退出方法之间的时间。 * 自定义指标——这使用自定义表达式根据代码中可用的值生成指标。在每种情况下，我们都会获得用于创建指标的标准 Lightrun 对话框。这让我们可以配置指标，包括为指标添加名称、触发条件以及停止工作的到期时间——与我们对快照和日志完全相同的方式。这些指标默认输出到日志记录进程，但如果需要，也可以集成到 StatsD、Prometheus 和其他工具中。 ### 4.1。计数器 计数器是对某些代码执行次数的简单度量。 每次到达我们的代码行时，计数器都会增加 1，然后我们可以看到这种情况发生的频率。添加计数器的方法是从 Lightrun 菜单中选择计数器，然后填写对话框： 其中大部分是相当标准的。唯一不寻常的是“名称”字段——我们需要给每个计数器一个唯一的名称，以便我们可以跟踪它们。特别是，这里的一个强大功能是我们可以设置计数器 - 就像所有指标一样 - 附加条件。这使我们能够计算仅在满足其他条件时达到某行代码的次数，例如仅针对某个用户或影响某些记录。添加计数器不会立即执行任何操作。但是，一旦第一次触发，它将开始以与日志输出方式类似的方式报告指标值——基于管道设置的日志输出。我们的编辑器每 10 秒显示一次指标的值：[](/wp-content/ uploads/2022/09/log-metrics.png) 而日志输出每秒显示指标：![](https://www.baeldung.com/wp-content/uploads/2022/09/log-output -metrics.png) 此输出由与日志输出完全相同的设置控制，并将我们的指标与我们的日志交错以提供更好的正在发生的事情的图片。这让我们可以随着时间的推移跟踪指标是如何变化的，以查看代码被触发的速率。 ### 4.2。 Time Duration 计数器用于衡量一行代码的执行次数，Time Duration 用于衡量代码运行的时间。 这些也称为“TicToc”指标——Tic开始录音，Toc 停止录音，类似于时钟的噪音。创建持续时间时，我们需要准确配置我们正在测量的代码段。这是通过指定开始和停止录制的行来完成的——通过在打开对话框之前选择一个代码块或在对话框中输入行号：![](https://www.baeldung.com /wp-content/uploads/2022/09/time-duration.png) 除此之外，创建持续时间与创建持续时间相同柜台。 一旦创建，这些指标将立即开始输出，而不是等待它们第一次被触发。 这些输出是插件和应用程序标准输出，与计数器完全相同，并会告诉我们多久代码部分已运行以及代码运行的最快、最慢和平均时间：![](https://www.baeldung.com/wp-content/uploads/2022/09/time- duration-metrics.png) ### 4.3。 Method Duration Method Duration 指标与 Time Duration 基本相同，但我们指定完整方法，而不是指定开始行和结束行：![](https://www.baeldung.com/wp-content/uploads/ 2022/09/method-duration.png) 添加后，此功能与覆盖整个方法主体的持续时间指标完全相同。开始时间认为是进入方法的时间，结束时间是离开的时间，不管是返回还是异常：![](https://www.baeldung.com/wp -content/uploads/2022/09/method-duration-logs.png) 从这里的输出我们可以看出，这些实际上是“ TicToc Log”条目也是如此，只有 Lightrun 根据方法本身而不是基于代码行自动为我们确定了起点和终点。 ### 4.4。自定义指标 我们的最终指标只是一个自定义指标。这让我们可以以某种方式聚合代码中存在的数字，无论这些数字是什么。 例如，我们可能想要计算某些搜索结果中返回的记录数。在创建这种类型的指标时，我们需要指定一个表达式。此表达式返回我们的指标将聚合的数字，与其他地方一样，它可以是我们代码中此时可以计算的任何表达式：![](https://www.baeldung.com/wp-content/uploads /2022/09/custom-metrics.png) 当这些指标输出时，它向我们展示了它被触发的次数和最大值、最小值，以及我们表达式的平均值：[](/wp-content/uploads/ 2022/09/custom-metrics-logs.png) 5。摘要 -------------- **这里我们详细介绍了 Lightrun 可以让我们更深入地了解我们的应用程序并更好地了解它们的工作原理的主要方式。**为什么不在您的下一个应用程序中使用它，以更好地了解正在发生的事情，甚至帮助诊断可能发生的任何问题？
