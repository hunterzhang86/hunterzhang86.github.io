Spring Security - 攻击 OAuth

 2023.1.23  2023.3.13  手册  3387  7 分钟  

一、简介

OAuth 是委托授权的行业标准框架。在创建构成标准的各种流程时，我们付出了很多思考和关注。即便如此，它也不是没有漏洞。

在本系列文章中，我们将从理论的角度讨论针对 OAuth 的攻击，并描述为保护我们的应用程序而存在的各种选项。

2.授权码授予

授权码授予 流程是大多数实现委托授权的应用程序使用的默认流程。

在该流程开始之前，Client 必须预先注册到 Authorization Server，并且在此过程中，它还必须提供一个重定向 URL --- 即一个 URL，Authorization Server 可以通过该 URL 回调到具有授权码的客户。

让我们仔细看看它是如何工作的以及其中一些术语的含义。

在授权代码授予流程中，客户端（请求委托授权的应用程序）将资源所有者（用户）重定向到授权服务器（例如，[使用 Google 登录](https://developers.google.com/identity /登录/网络/登录））。登录后，授权服务器使用授权码重定向回客户端。

接下来，客户端调用授权服务器的端点，通过提供授权代码请求访问令牌。至此，流程结束，客户端可以使用令牌访问授权服务器保护的资源。

现在，OAuth 2.0 框架允许这些客户端是公开的，例如在客户端无法安全地持有客户端密钥的情况下。让我们看一下可能针对公共客户端的一些重定向攻击。

3.重定向攻击

3.1。攻击前提

重定向攻击依赖于这样一个事实：**OAuth 标准没有完全描述必须指定此重定向 URL 的程度。**这是设计使然。

这允许 OAuth 协议的某些实现允许部分重定向 URL。

例如，如果我们使用以下基于通配符的匹配对授权服务器注册客户端 ID 和客户端重定向 URL：

*.cloudapp.net

这将适用于：

app.cloudapp.net

但也适用于：

evil.cloudapp.net

我们特意选择了 cloudapp.net 域，因为这是我们可以托管 OAuth 支持的应用程序的真实位置。该域是 Microsoft 的 Windows Azure 平台 的一部分-environment-in-windows-azure) 并允许任何开发人员在其下托管子域来测试应用程序。这本身不是问题，但它是更大利用的重要组成部分。

该漏洞利用的第二部分是一个授权服务器，它允许在回调 URL 上进行通配符匹配。

最后，为了实现这个漏洞，应用程序开发者需要向授权服务器注册以接受主域下的任何 URL，格式为 *.cloudapp.net。

3.2。攻击

当满足这些条件时，攻击者需要诱使用户从他控制的子域启动页面，例如，发送[用户看起来真实的电子邮件]（https://www.vadesecure.com/en /5-common-phishing-techniques/) 要求他对受 OAuth 保护的帐户采取一些措施。通常，这看起来像 https://evil.cloudapp.net/login。当用户打开此链接并选择登录时，他将通过授权请求重定向到授权服务器：

1

GET /authorize?response_type=code&client_id={apps-client-id}&state={state}&redirect_uri=https%3A%2F%2Fevil.cloudapp.net%2Fcb HTTP/1.1

虽然这看起来很典型，但此 URL 是恶意的。请看，在这种情况下，授权服务器 ** 收到一个带有 *app 的 * 客户端 ID ** 的篡改 URL 和一个指向 *evil 的 * 应用的重定向 URL。

然后，授权服务器将验证 URL，该 URL 是指定主域下的子域。由于授权服务器认为请求来自有效来源，因此它将验证用户身份，然后像往常一样请求同意。

完成此操作后，它现在将重定向回 evil.cloudapp.net 子域，将授权码交给攻击者。

由于攻击者现在有了授权码，他需要做的就是使用授权码调用授权服务器的令牌端点来接收令牌，这允许他访问资源所有者的受保护资源。

但是，请注意，我们也在此处配置重定向 URL，并且它是绝对的。我们可以通过这样做来减轻漏洞。

4.1。严格的

默认情况下，Spring OAuth 在重定向 URL 匹配方面允许一定程度的灵活性。

例如，DefaultRedirectResolver支持子域匹配。

让我们只使用我们需要的。 如果我们可以完全匹配重定向 URL，我们应该这样做：

@配置

在这种情况下，我们已切换到使用 ExactMatchRedirectResolver 来处理重定向 URL。此解析器执行精确的字符串匹配，而不以任何方式解析重定向 URL。 这使其行为更加安全和确定。

   return schemeMatch && userInfoMatch && hostMatch && portMatch && pathMatch && queryParamMatch;

我们可以看到 URL 匹配是通过将传入的重定向 URL 解析为其组成部分来完成的。这是相当复杂的，因为它有几个特性，**比如端口、子域和查询参数是否应该匹配。**并且选择允许子域匹配是需要三思而后行的事情。

当然，这种灵活性是存在的，如果我们需要它——让我们谨慎使用它。

  1. 隐式流重定向攻击

需要明确的是，隐式流程 [不推荐](https://tools.ietf.org/html/rfc8252#第 8.2 节）。 最好使用授权代码授予流程以及 [PKCE]（https://tools.ietf.org/html/rfc7636）提供的额外安全性。也就是说，让我们看一下重定向攻击如何通过隐式流表现出来。

针对隐式流的重定向攻击将遵循与我们在上面看到的相同的基本轮廓。主要区别在于攻击者立即获得令牌，因为没有授权码交换步骤。

和以前一样，重定向 URL 的绝对匹配也将减轻此类攻击。

此外，我们可以发现隐式流包含另一个相关的漏洞。 攻击者可以将客户端用作打开的重定向器并让它重新附加片段。

攻击和以前一样开始，攻击者让用户访问攻击者控制下的页面，例如 https://evil.cloudapp.net/info。该页面旨在像以前一样发起授权请求。但是，它现在包含一个重定向 URL：

1
2

GET /authorize?response_type=token&client_id=ABCD&state=xyz&redirect_uri=https%3A%2F%2Fapp.cloudapp.net%2Fcb%26redirect_to
%253Dhttps%253A%252F%252Fevil.cloudapp.net%252Fcb HTTP/1.1

redirect_to https://evil.cloudapp.net 正在设置授权端点以将令牌重定向到攻击者控制的域。授权服务器现在将首先重定向到实际的应用程序站点：

1

位置：https://app.cloudapp.net/cb?redirect_to%3Dhttps%3A%2F%2Fevil.cloudapp.net%2Fcb#access_token=LdKgJIfEWR34aslkf&...

当这个请求到达打开的重定向器时，它会提取重定向 URL evil.cloudapp.net，然后重定向到攻击者的站点：

1

https://evil.cloudapp.net/cb#access_token=LdKgJIfEWR34aslkf&...

绝对 URL 匹配也将减轻这种攻击。

6.总结

在本文中，我们讨论了基于重定向 URL 的针对 OAuth 协议的一类攻击。

虽然这可能会产生严重后果，但在授权服务器上使用绝对 URL 匹配可以减轻此类攻击。 安全底
